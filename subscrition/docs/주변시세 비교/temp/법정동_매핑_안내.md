# 법정동 매핑 입력 안내

## 한 줄 요약

법정동 매핑은 코드에 하드코딩하는 것이 아니라 `lawd_code_mapping` 테이블에 넣고, 이후 공고 unit 주소를 재정규화해서 `AnnouncementUnit.legalDongCode`와 `AnnouncementUnit.lawdCd`를 채우는 구조다.

## 핵심 개념

| 항목 | 설명 |
| - | - |
| 저장 테이블 | `lawd_code_mapping` |
| 조회 기준 | `regionLevel2 + legalDongName + active=true` |
| 법정동 코드 | `legalDongCode`, 10자리 숫자 |
| RTMS 지역 코드 | `lawdCd`, `legalDongCode` 앞 5자리에서 자동 생성 |
| unit 저장 필드 | `AnnouncementUnit.legalDongCode`, `AnnouncementUnit.lawdCd`, `addressStatus` |

예시:

| regionLevel1 | regionLevel2 | legalDongName | legalDongCode | lawdCd |
| - | - | - | - | - |
| 인천광역시 | 부평구 | 부평동 | `2823710100` | `28237` |
| 인천광역시 | 남동구 | 논현동 | `2820011000` | `28200` |

## 넣는 방법 1: 공공 법정동 코드 seed import

최신 공공 법정동 코드 CSV/TSV 원문을 확보한 뒤 관리자 API로 import한다.

```http
POST /api/admin/market/address/lawd-code-mappings/import
Authorization: Bearer {ADMIN_TOKEN}
Content-Type: application/json
```

```json
{
  "delimiter": "tab",
  "activeOnly": true,
  "content": "법정동코드\t법정동명\t폐지여부\n2823710100\t인천광역시 부평구 부평동\t존재"
}
```

성공 기준:

```text
parsedCount > 0
insertedCount + updatedCount > 0
```

주의사항:

- `content`에는 공공 법정동 코드 CSV/TSV 파일 내용을 그대로 넣는다.
- header는 `법정동코드`, `법정동명`, `폐지여부`를 우선 인식한다.
- `activeOnly=true`면 폐지 row는 skip한다.
- repo 안에는 공공 법정동 코드 전체 원본 파일이 없다. 개발/운영 DB마다 최신 파일을 별도로 import해야 한다.

## 넣는 방법 2: 특정 매핑만 수동 upsert

특정 법정동만 추가하거나 수정할 때는 수동 upsert API를 쓴다.

```http
POST /api/admin/market/address/lawd-code-mappings
Authorization: Bearer {ADMIN_TOKEN}
Content-Type: application/json
```

```json
{
  "mappings": [
    {
      "regionLevel1": "인천광역시",
      "regionLevel2": "부평구",
      "legalDongName": "부평동",
      "legalDongCode": "2823710100",
      "active": true
    }
  ]
}
```

`legalDongCode`는 반드시 10자리 숫자여야 한다. `lawdCd`는 `2823710100`의 앞 5자리인 `28237`로 자동 계산된다.

## 매핑 후 기존 unit 재정규화

매핑을 넣은 뒤 기존 공고 unit에 `lawdCd`를 채우려면 재정규화 API를 호출한다.

```http
POST /api/admin/market/address/announcements/{announcementId}/normalize-units
Authorization: Bearer {ADMIN_TOKEN}
Content-Type: application/json
```

```json
{
  "retryNoLawdCode": true
}
```

- `retryNoLawdCode=false` 또는 생략 시 `NOT_REQUESTED` unit만 처리한다.
- `retryNoLawdCode=true`면 기존 `NO_LAWD_CODE` unit도 다시 처리한다.
- LH import/reimport 후 신규 unit은 자동으로 주소 정규화를 시도한다.

확인 SQL 예시:

```sql
select id,
       announcement_id,
       full_address,
       normalized_address,
       legal_dong_code,
       lawd_cd,
       address_status,
       address_message,
       exclusive_area_value
from announcement_unit
where announcement_id = {announcementId}
order by id;
```

성공 기준:

```text
비교 대상 unit의 address_status = SUCCESS
비교 대상 unit의 legal_dong_code is not null
비교 대상 unit의 lawd_cd is not null
비교 대상 unit의 exclusive_area_value is not null
```

## 내부 동작 흐름

```text
LH import/reimport
→ AnnouncementUnit 저장
→ AddressNormalizationService가 주소에서 regionLevel2/legalDongName 추출
→ lawd_code_mapping에서 active=true 매핑 조회
→ 매핑 성공 시 legalDongCode/lawdCd 저장
→ RTMS 수집 및 snapshot 집계
→ 주변시세 비교 API 호출
```

주변시세 비교 API는 `unit.lawdCd`가 없으면 `UNIT_LAWD_CD_MISSING` 상태로 비교를 중단한다. 따라서 법정동 매핑과 unit 재정규화가 선행되어야 한다.

## 현재 한계

| 케이스 | 현재 처리 |
| - | - |
| `인천광역시 부평구 부평동 297` | `부평구 + 부평동`으로 매핑 가능 |
| `경기도 수원시 영통구 이의동 1` | 시와 구가 같이 있으면 더 구체적인 `영통구` 사용 |
| `인천광역시 남동구 은봉로 297(논현동)` | 괄호 안 `논현동`을 법정동으로 우선 사용 |
| `인천광역시 남동구 은봉로 297` | 괄호 법정동이 없으면 현재 확정 실패 가능 |

괄호 법정동이 없는 도로명주소는 `NO_LAWD_CODE`가 남을 수 있다. 이 경우 Juso API 또는 geocoding 기반 법정동 보강이 후속 과제다.
