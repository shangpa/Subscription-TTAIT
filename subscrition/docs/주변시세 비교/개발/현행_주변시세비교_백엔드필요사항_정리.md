# [현행] 주변시세 비교 백엔드 필요사항 정리

## 1. 목적

공고 상세 화면의 주변시세 비교를 실제 데이터로 표시하기 위해 backend에서 준비하거나 정리해야 할 항목만 정리한다.

이 문서는 frontend 화면 구성이나 디자인 요구사항을 다루지 않는다.

## 2. 현재 결론

주변시세 비교 API와 관리자 운영 API는 구현되어 있다. 다만 현재 local/runtime 데이터 기준으로는 선행 데이터가 없어 사용자 화면에서 비교값이 나오지 않는다.

현재 확인된 blocker는 아래 순서다.

```text
lawd_code_mapping = 0건
announcement_unit.lawd_cd = null
→ public 비교 API status = UNIT_LAWD_CD_MISSING
→ snapshot 조회 단계로 진입하지 않음
```

추가로 아래 테이블도 비어 있어, `lawdCd`를 채운 뒤에도 RTMS 수집과 snapshot 집계가 필요하다.

```text
market_transaction_raw = 0건
market_price_snapshot = 0건
```

## 3. 현재 구현된 흐름

사용자 public API는 데이터를 생성하지 않고 이미 만들어진 snapshot만 조회한다.

```text
관리자/운영 선처리
→ 법정동 seed import
→ unit 주소 정규화
→ RTMS raw 수집
→ market snapshot 집계
→ 사용자 public API에서 비교 조회
```

public API:

```http
GET /api/announcements/{announcementId}/units/{unitId}/market-comparison?sourceType=APT_RENT&dealYmFrom=YYYYMM&dealYmTo=YYYYMM
```

관리자 API는 모두 `ADMIN` 권한이 필요하다.

```text
Authorization: Bearer {ADMIN_TOKEN}
```

## 4. backend 운영 실행 순서

### 4.1 ADMIN token 준비

기존 관리자 계정이 있으면 login 후 `accessToken`을 사용한다.

```http
POST /api/auth/login
Content-Type: application/json
```

```json
{
  "loginId": "admin",
  "password": "password"
}
```

관리자 계정이 없으면 회원가입 후 DB에서 role을 변경하고 다시 login해야 한다.

```sql
update users
set role = 'ADMIN'
where login_id = '{loginId}';
```

### 4.2 법정동 seed import

현재 가장 먼저 필요한 단계다.

```http
POST /api/admin/market/address/lawd-code-mappings/import
Authorization: Bearer {ADMIN_TOKEN}
Content-Type: application/json
```

```json
{
  "delimiter": "tab",
  "activeOnly": true,
  "content": "법정동코드\t법정동명\t폐지여부\n4423011000\t충청남도 논산시 강경읍\t존재"
}
```

성공 기준:

```text
parsedCount > 0
insertedCount + updatedCount > 0
```

정리 필요:

- repo 안에는 공공 법정동 코드 전체 원본 파일이 없다.
- backend/운영 담당자가 최신 법정동 CSV/TSV 원본 확보 위치를 정해야 한다.
- 개발/운영 DB 초기화 시 seed import 절차를 문서화해야 한다.

### 4.3 공고 unit 주소 재정규화

```http
POST /api/admin/market/address/announcements/{announcementId}/normalize-units
Authorization: Bearer {ADMIN_TOKEN}
Content-Type: application/json
```

```json
{
  "retryNoLawdCode": true
}
```

성공 기준:

```text
비교 대상 unit의 address_status = SUCCESS
비교 대상 unit의 legal_dong_code is not null
비교 대상 unit의 lawd_cd is not null
비교 대상 unit의 exclusive_area_value is not null
```

확인 SQL:

```sql
select id, announcement_id, full_address, normalized_address,
       legal_dong_code, lawd_cd, address_status, address_message,
       address_normalized_at, exclusive_area_value
from announcement_unit
where announcement_id = {announcementId}
order by id;
```

정리 필요:

- `NO_LAWD_CODE`가 남으면 법정동 seed 부족인지 주소 parsing 문제인지 backend에서 구분 가능한 메시지가 필요하다.
- `NO_ADDRESS` unit은 주소 원천 데이터 보강 없이는 자동 정규화가 불가능하다.
- `UNIT_AREA_MISSING` unit은 시세 비교 대상에서 제외하거나 unit 추출/검수 단계에서 보정해야 한다.

### 4.4 RTMS 수집과 snapshot 집계

`lawdCd`가 채워진 뒤 실행한다.

```http
POST /api/admin/market/batches/rtms-snapshot
Authorization: Bearer {ADMIN_TOKEN}
Content-Type: application/json
```

```json
{
  "sourceType": "APT_RENT",
  "lawdCd": "44230",
  "dealYm": "202606",
  "numOfRows": 100,
  "maxPages": 10,
  "dealYmFrom": "202601",
  "dealYmTo": "202606",
  "areaMin": 35.0,
  "areaMax": 85.0,
  "minimumSampleCount": 3
}
```

성공 기준:

```text
status = SUCCESS
snapshotAggregated = true
snapshot.status = OK
```

주의:

- `RTMS_SERVICE_KEY`가 없으면 수집이 실패한다.
- smoke test에서는 `minimumSampleCount=1`을 쓸 수 있지만 운영 기준은 별도 확정이 필요하다.
- `dealYmFrom/dealYmTo`는 public API 조회값과 정확히 같아야 snapshot이 매칭된다.
- `areaMin <= unit.exclusive_area_value <= areaMax` 조건을 만족해야 snapshot이 매칭된다.

## 5. backend에서 frontend admin 연동 전에 보강하면 좋은 API/DTO

현재 frontend admin 화면에서 시장 데이터 운영 기능을 붙이려면 아래 값이 필요하다. 일부 값은 public detail 응답이나 현 admin DTO에 충분히 노출되어 있지 않다.

### 5.1 admin unit 응답 보강

관리자 화면에서 DB를 직접 보지 않고 처리하려면 `AdminAnnouncementUnitResponse` 또는 별도 admin market readiness API에 아래 필드가 필요하다.

```text
legalDongCode
lawdCd
addressStatus
addressMessage
addressNormalizedAt
exclusiveAreaValue
depositAmount
monthlyRentAmount
salePriceMin
salePriceMax
```

현재 admin unit 응답에는 `lawdCd`, `legalDongCode`, `addressStatus`가 없다. 이 상태에서는 frontend admin 화면이 batch parameter를 자동으로 만들기 어렵다.

### 5.2 readiness 조회 API 검토

운영자가 공고별 준비 상태를 한 번에 볼 수 있도록 아래 형태의 admin read API가 있으면 좋다.

```http
GET /api/admin/market/announcements/{announcementId}/readiness
```

필요 응답 예시:

```json
{
  "announcementId": 47,
  "units": [
    {
      "unitId": 30,
      "fullAddress": "충남 논산시 강경읍 동안로 96",
      "lawdCd": "44230",
      "addressStatus": "SUCCESS",
      "exclusiveAreaValue": 39.0,
      "recommendedSourceType": "APT_RENT",
      "marketReady": false,
      "blocker": "SNAPSHOT_NOT_FOUND"
    }
  ]
}
```

이 API가 없으면 frontend admin 화면은 여러 API와 DB 상태를 조합해야 한다.

### 5.3 announcement 단위 batch 지원 검토

현재 batch API는 `lawdCd`, `dealYm`, `areaMin`, `areaMax`를 호출자가 직접 넣어야 한다. 운영 UI를 단순화하려면 backend가 공고 또는 unit 기준으로 필요한 batch parameter를 계산해주는 endpoint가 필요하다.

예시:

```http
POST /api/admin/market/announcements/{announcementId}/prepare
```

요청 예시:

```json
{
  "dealYmFrom": "202601",
  "dealYmTo": "202606",
  "minimumSampleCount": 3
}
```

backend 처리 기대:

```text
unit별 lawdCd/exclusiveAreaValue 확인
→ sourceType 추천
→ 필요한 lawdCd/dealYm/area 범위별 RTMS batch 실행
→ snapshot 결과 요약 반환
```

## 6. 정리해야 하는 항목

### 6.1 데이터 운영 정리

- 법정동 seed 원본 파일 확보 위치와 갱신 주기.
- local/dev/prod DB 초기화 시 seed import 절차.
- `RTMS_SERVICE_KEY` 설정 위치와 누락 시 대응.
- RTMS 수집 기간 기본값.
- snapshot 면적 범위 산정 기준.
- `minimumSampleCount` 운영 기준.

### 6.2 API 계약 정리

- public comparison API의 `dealYmFrom/dealYmTo` 기본 정책.
- sourceType 추천 기준.
- admin readiness 조회 필요 여부.
- announcement 단위 prepare endpoint 필요 여부.
- `NO_LAWD_CODE`, `NO_ADDRESS`, `UNIT_AREA_MISSING`, `SNAPSHOT_NOT_FOUND`, `INSUFFICIENT_DATA`별 backend 메시지 정책.

### 6.3 보안/운영 정리

- `/api/admin/**` 호출은 admin 권한 JWT만 허용.
- 문서나 설정 파일에 남은 credential성 값은 제거하거나 rotate한다.
- `.env`, DB password, API key, JWT secret은 응답/로그/커밋에 노출하지 않는다.
- hardcoded service key가 있으면 환경변수로 이전한다.

### 6.4 자동화 정리

현재 구현은 수동 admin API 실행 방식이다.

없는 것:

```text
market RTMS/snapshot scheduler
progress table
async job id
polling endpoint
public API에서 batch trigger
```

운영 자동화가 필요하면 scheduler 또는 admin prepare job을 별도 구현해야 한다.

## 7. backend에 전달할 요약

```text
시세비교 UI/API 연결은 되어 있지만 실데이터 비교는 backend 운영 데이터가 비어 있어 막혀 있다.

현재 blocker는 lawd_code_mapping 0건과 announcement_unit.lawd_cd null이다.
먼저 법정동 seed import와 normalize-units가 필요하고,
그 다음 RTMS_SERVICE_KEY가 설정된 상태에서 rtms-snapshot batch를 실행해야 한다.

frontend admin 연동을 위해서는 admin unit 응답에 lawdCd/addressStatus/legalDongCode를 추가하거나,
공고별 market readiness/prepare API를 제공하는 것이 필요하다.
```
