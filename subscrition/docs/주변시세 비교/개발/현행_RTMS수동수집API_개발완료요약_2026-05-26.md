# [현행] RTMS 수동 수집 API 개발 완료 요약 2026-05-26

<!-- markdownlint-disable MD013 -->

## 1. 작성 목적

이 문서는 `feature/seuil/price_comparison` branch에서 관리자 RTMS 수동 수집 API와 raw transaction 저장 service 구현 완료 범위를 정리한다.
주변시세 비교 전체 기능 완료 문서가 아니라, `RtmsClient` 결과를 `MarketTransactionRaw`로 저장하는 backend 연결 단계의 완료 요약이다.

## 2. 완료 범위

| 영역 | 완료 내용 |
| - | - |
| 관리자 API | `POST /api/admin/market/rtms/collect` 추가 |
| request DTO | `sourceType`, `lawdCd`, `dealYm`, `pageNo`, `numOfRows` 수신 |
| request default | `pageNo=1`, `numOfRows=100` 기본값 적용 |
| 수집 service | `RtmsClient.fetch(...)` 단일 page 호출 |
| raw 저장 | `RtmsTransactionItem`을 `MarketTransactionRaw`로 변환 저장 |
| 중복 방지 | `rawPayloadHash` 기준 중복 저장 skip |
| hash 기준 | `sourceType|lawdCd|dealYm|rawPayload` 조합의 SHA-256 |
| 상태 응답 | `SUCCESS`, `NO_RESULT`, `FAILED` 기준 count 응답 |
| 테스트 | service, admin service, controller, DTO JSON test 추가 |

## 3. API 계약

### 3.1 Endpoint

```http
POST /api/admin/market/rtms/collect
```

`/api/admin/**` 경로이므로 기존 `SecurityConfig` 기준 ADMIN 권한 보호 대상이다.

### 3.2 Request 예시

```json
{
  "sourceType": "APT_RENT",
  "lawdCd": "41570",
  "dealYm": "202405",
  "pageNo": 1,
  "numOfRows": 100
}
```

| 필드 | 설명 | 비고 |
| - | - | - |
| `sourceType` | RTMS source type | 예: `APT_RENT`, `APT_TRADE` |
| `lawdCd` | 법정동 코드 앞 5자리 | `RtmsClient`에서 5자리 검증 |
| `dealYm` | 계약년월 | `RtmsClient`에서 `YYYYMM` 검증 |
| `pageNo` | 조회 page 번호 | 생략 시 `1` |
| `numOfRows` | page당 조회 건수 | 생략 시 `100` |

### 3.3 Response 예시

```json
{
  "sourceType": "APT_RENT",
  "lawdCd": "41570",
  "dealYm": "202405",
  "status": "SUCCESS",
  "fetchedCount": 2,
  "savedCount": 1,
  "duplicateCount": 1,
  "failedCount": 0,
  "message": null
}
```

| 필드 | 의미 |
| - | - |
| `status` | `RtmsApiResult.Status` 이름 |
| `fetchedCount` | 이번 page에서 adapter가 반환한 item 수 |
| `savedCount` | 신규 저장된 raw transaction 수 |
| `duplicateCount` | `rawPayloadHash` 중복으로 skip한 수 |
| `failedCount` | 요청 단위 실패 수. `FAILED`이면 `1` |
| `message` | RTMS adapter/client 결과 메시지 |

## 7. 남은 리스크

| 리스크 | 설명 | 후속 방향 |
| - | - | - |
| `resultCode=03 No Data` 정책 | 현재 `RtmsResponseAdapter`는 `resultCode != 00`을 `FAILED`로 유지한다. | `03`을 `NO_RESULT`로 볼지 정책 결정 후 adapter/test 수정 |
| 전체 pagination | 현재 API는 단일 page만 수집한다. | `totalCount`, `pageNo`, `numOfRows` 기반 전체 순회 정책 설계 |
| 중복 race | `existsByRawPayloadHash` 후 `save` 사이 동시 요청 race 가능성이 있다. | DB unique constraint 기반 duplicate key 예외 처리 보강 |
| transaction 범위 | 현재 수집 service는 fetch와 저장이 같은 service method 안에 있다. | 대량 수집 전 fetch와 저장 transaction 분리 검토 |

## 8. 후속 작업

1. RTMS `resultCode=03 No Data` 처리 정책 확정.
2. `totalCount` 기반 전체 pagination 수집 API 또는 batch 설계.
3. `MarketTransactionRaw` 기반 `MarketPriceSnapshot` aggregation 구현.
4. public/admin 주변시세 비교 API 설계와 구현.
5. frontend 지도/리스트 UI 연결. 이 항목은 다른 팀원 담당이며 이번 작업 범위 밖이다.

## 9. 제외 범위

- 실제 RTMS API key 추가 또는 secret 커밋.
- `RtmsResponseAdapter`의 `resultCode` 정책 변경.
- `MarketTransactionRaw` entity 구조 변경.
- public 주변시세 비교 API.
- snapshot aggregation.
- frontend 지도/리스트 UI 수정.
