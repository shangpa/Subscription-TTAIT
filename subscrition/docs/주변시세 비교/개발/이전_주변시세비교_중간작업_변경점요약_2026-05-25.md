# [이전] 주변시세 비교 중간 작업 변경점 요약 2026-05-25

<!-- markdownlint-disable MD013 -->

## 1. 작성 목적

이 문서는 `feature/seuil/price_comparison` branch에서 2026-05-25 기준으로 커밋 직전에 남아 있는 주변시세 비교 관련 변경점을 정리한다.
작업 세션이 길어져 중단 지점을 놓치기 쉬운 상태였기 때문에, 집 PC에서 이어서 작업할 때 현재 구현 범위와 다음 작업을 빠르게 확인하기 위한 인계 문서로 사용한다.

## 2. 현재 작업 상태

| 구분 | 상태 | 메모 |
| - | - | - |
| 좌표/Geocoding 1차 MVP | 완료 유지 | `AnnouncementUnit` 좌표, geocoding 상태, admin/public DTO 노출 기반 |
| 상가 공고 skip 정책 | 구현 반영 | LH import candidate와 dedupe/import 흐름에서 상가 공고 제외 기준 추가 |
| 주소/법정동 기반 | 1차 구현 | 주소 정규화 서비스와 `AddressResolutionStatus` 추가 |
| RTMS client/config/adapter | 1차 구현 | 국토부 RTMS API 호출 client, properties, XML adapter 추가 |
| market raw/snapshot domain/repository | 1차 구현 | 실거래 raw 저장과 snapshot 저장용 entity/repository/test 추가 |
| 관리자 RTMS 수동 수집 API | 구현 완료 | `POST /api/admin/market/rtms/collect`, raw transaction 저장 service/test 추가 |
| 주변시세 비교 API | 미구현 | 아직 controller/service 응답 API 단계 아님 |

현재 코드는 주변시세 비교 전체 기능 완성이 아니라, RTMS 수집과 market 저장 구조를 붙이기 위한 backend foundation 단계다.

## 3. 주요 변경 영역

### 3.1 LH import 후보와 상가 제외 정책

- `LhImportCandidate`에 상가 공고 여부, hash, 수집 시각 등 import 후보 검토에 필요한 필드를 확장했다.
- `LhImportDecisionType`에 상가 skip 유형을 추가했다.
- `LhImportDedupeDecisionService`, `NoticeImportOrchestrator`, `AdminLhImportManagementService`에 상가 공고 skip 결과가 반영되도록 보강했다.
- admin DTO와 테스트에서 상가 여부, skip 사유, hash 값이 노출/검증되도록 수정했다.

### 3.2 주소/법정동 기반

- `AddressResolutionStatus`를 추가해 unit 주소 처리 상태를 분리했다.
- `AnnouncementUnit`에 주소 해석 상태와 법정동 코드 관련 필드를 추가했다.
- `AddressNormalizationService`를 추가해 공고 주소를 주변시세 조회용 주소 기반으로 정규화하는 1차 로직을 만들었다.

### 3.3 RTMS client/config/adapter

- `external/rtms` package를 추가했다.
- `RtmsProperties`, `RtmsConfig`, `RtmsClient`를 통해 국토부 RTMS API 호출 기반을 만들었다.
- `RtmsResponseAdapter`에서 XML 응답을 `RtmsApiResult`, `RtmsTransactionItem`으로 변환한다.
- XML parser는 `DOCTYPE`, external entity를 비활성화해 XXE 위험을 줄이도록 설정했다.
- `application.properties`에는 실제 key가 아니라 `rtms.service-key=${RTMS_SERVICE_KEY:}` placeholder만 추가했다.

### 3.4 market 저장 구조

- `market/domain`에 아래 entity와 enum을 추가했다.
  - `MarketTransactionRaw`
  - `MarketPriceSnapshot`
  - `LawdCodeMapping`
  - `MarketSourceType`
  - `MarketTransactionType`
  - `MarketSnapshotStatus`
- `market/repository`에 raw transaction, snapshot, 법정동 mapping repository를 추가했다.
- raw payload hash와 snapshot key 기준 중복 방지 제약을 둬 반복 수집 시 같은 데이터가 중복 저장되지 않도록 했다.

### 3.5 관리자 RTMS 수동 수집 API

- `POST /api/admin/market/rtms/collect` endpoint를 추가했다.
- `MarketRtmsCollectionService`에서 `RtmsClient.fetch(...)` 결과를 `MarketTransactionRaw`로 저장한다.
- `rawPayloadHash`는 `sourceType|lawdCd|dealYm|rawPayload` 조합의 SHA-256으로 생성한다.
- 같은 `rawPayloadHash`가 이미 있으면 저장하지 않고 `duplicateCount`로 집계한다.
- `NO_RESULT`는 저장 없음/실패 아님, `FAILED`는 저장 없음/`failedCount=1`로 응답한다.
- 단일 page fetch만 지원하며 `totalCount` 기반 전체 pagination 순회는 후속으로 남겼다.

## 4. 검증 결과

### 4.1 통과한 Gradle focused test

아래 명령으로 변경 관련 focused test를 실행했고 통과했다.


### 4.2 LSP 진단 참고

Java LSP 진단은 Lombok getter/builder를 인식하지 못해 `getXxx() is undefined`, `builder() is undefined` 계열 오류를 다수 표시했다.
동일 파일들이 Gradle compile/test에서는 통과했으므로, 현재 커밋 기준 검증은 Gradle 결과를 우선한다.

## 5. 다음 작업 후보

1. RTMS `resultCode=03 No Data`를 `NO_RESULT`로 볼지 정책을 정한다.
2. `totalCount` 기반 전체 pagination 순회와 수집 범위를 정한다.
3. raw transaction 기반으로 `MarketPriceSnapshot`을 생성하는 aggregation 정책을 정한다.
4. 주변시세 비교 public/admin API 응답 계약을 설계한다.
5. 마지막에 frontend Dynamic Map과 marker 연동은 별도 UI 작업으로 분리한다.

## 6. 주의할 점

- 이 커밋에는 실제 RTMS service key가 포함되지 않는다.
- 현재 `market` package는 raw 저장 구조와 관리자 수동 수집 API까지 포함한다. 사용자에게 노출되는 주변시세 비교 API는 아직 없다.
- 집 PC에서 이어서 작업할 때는 먼저 branch를 pull한 뒤 `subscrition`에서 focused test 또는 전체 test를 다시 실행한다.
