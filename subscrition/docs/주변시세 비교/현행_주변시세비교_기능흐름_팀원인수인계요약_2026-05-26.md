# [현행] 주변시세 비교 기능 흐름 팀원 인수인계 요약 2026-05-26

<!-- markdownlint-disable MD013 -->

## 1. 이 문서를 먼저 읽는 사람

이 문서는 주변시세 비교 기능을 이어받는 팀원과 팀원 AI를 위한 요점정리 문서다.
현재 `main` 기준 구현 흐름, endpoint, DB table, service 위치, 선행 작업, 남은 리스크만 빠르게 파악하는 것을 목표로 한다.

## 2. 한 줄 요약

주변시세 비교는 프론트가 실시간 RTMS를 직접 호출하는 기능이 아니다.
관리자 흐름에서 `RTMS raw 거래 → MarketPriceSnapshot`을 미리 만들고, public API는 공고 `AnnouncementUnit`의 가격과 가장 맞는 snapshot 평균값을 비교해서 응답한다.

```text
LH 공고 수집
→ AnnouncementUnit 저장
→ unit 주소 정규화 + lawdCd 확보
→ RTMS 실거래 raw 수집
→ 법정동/월/면적 범위별 snapshot 집계
→ public 비교 API 호출
→ unit 가격과 snapshot 평균 차이 응답
```

## 6. 관리자 선행 작업 순서

### 6.1 법정동 mapping seed

```http
POST /api/admin/market/address/lawd-code-mappings/import
```

```json
{
  "delimiter": "tab",
  "activeOnly": true,
  "content": "법정동코드\t법정동명\t폐지여부\n2823710100\t인천광역시 부평구 부평동\t존재"
}
```

- header는 `법정동코드`, `법정동명`, `폐지여부`를 우선 인식한다.
- `activeOnly=true`면 폐지 row는 건너뛴다.
- 공공 전체 원본 파일은 repo에 넣지 않았다. 운영/개발 DB마다 최신 파일 content를 import한다.

### 6.2 LH unit 주소 재정규화

LH import 시 신규 unit은 자동으로 주소 정규화를 시도한다.
기존 `NO_LAWD_CODE` unit을 다시 처리하려면 아래 API를 쓴다.

```http
POST /api/admin/market/address/announcements/{announcementId}/normalize-units
```

```json
{
  "retryNoLawdCode": true
}
```

정상 처리되면 `announcement_unit`에 다음 값이 채워진다.

```text
legal_dong_code = 2823710100
lawd_cd = 28237
address_status = SUCCESS
```

### 6.3 RTMS raw 수집

단일 page:

```http
POST /api/admin/market/rtms/collect
```

전체 page:

```http
POST /api/admin/market/rtms/collect-all
```

```json
{
  "sourceType": "APT_RENT",
  "lawdCd": "28237",
  "dealYm": "202405",
  "numOfRows": 100,
  "maxPages": 10
}
```

`collect-all`은 `totalCount` 기준으로 중단하고, metadata가 없으면 현재 page item 수가 `numOfRows`보다 작은 시점을 마지막 page로 본다.

### 6.4 snapshot 집계

```http
POST /api/admin/market/snapshots/aggregate
```

```json
{
  "sourceType": "APT_RENT",
  "lawdCd": "28237",
  "dealYmFrom": "202405",
  "dealYmTo": "202405",
  "areaMin": 20.0,
  "areaMax": 30.0,
  "minimumSampleCount": 3
}
```

집계는 `market_transaction_raw`에서 같은 `sourceType`, `lawdCd`, 거래월 범위, 면적 범위 row를 조회한다.
기존 같은 조건의 snapshot은 `snapshotKey`로 찾아 삭제 후 새로 저장한다.

### 6.5 batch API

운영 편의상 수집과 집계를 한 번에 할 수 있다.

```http
POST /api/admin/market/batches/rtms-snapshot
```

```json
{
  "sourceType": "APT_RENT",
  "lawdCd": "28237",
  "dealYm": "202405",
  "numOfRows": 100,
  "maxPages": 10,
  "dealYmFrom": "202405",
  "dealYmTo": "202405",
  "areaMin": 20.0,
  "areaMax": 30.0,
  "minimumSampleCount": 3
}
```

내부 흐름:

```text
MarketRtmsCollectionService.collectAll
→ MarketPriceSnapshotAggregationService.aggregate
```

RTMS 수집이 `FAILED`면 snapshot 집계는 실행하지 않고 `COLLECTION_FAILED`를 반환한다.

## 7. 프론트가 호출할 public API

```http
GET /api/announcements/{announcementId}/units/{unitId}/market-comparison?sourceType=APT_RENT&dealYmFrom=202405&dealYmTo=202405
```

프론트는 이 API만 호출하면 된다.
RTMS 수집, 법정동 seed, snapshot 집계 API는 관리자/운영 선행 작업이다.

## 8. public 비교 API 내부 판단

`MarketComparisonService.compare()` 기준 흐름은 다음과 같다.

```text
1. announcementId, unitId, sourceType, dealYmFrom, dealYmTo 검증
2. unit 조회
3. unit.lawdCd 없으면 UNIT_LAWD_CD_MISSING
4. unit.exclusiveAreaValue 없으면 UNIT_AREA_MISSING
5. sourceType + lawdCd + dealYmFrom/To + 면적 포함 snapshot 조회
6. snapshot 없으면 SNAPSHOT_NOT_FOUND
7. snapshot.status != OK면 INSUFFICIENT_DATA
8. OK면 COMPARABLE + 보증금/월세/거래가 차이 계산
```

면적 매칭은 현재 `areaMin <= unit.exclusiveAreaValue <= areaMax` 조건이다.

## 9. status 의미

| Status | 의미 | 프론트 처리 제안 |
| - | - | - |
| `COMPARABLE` | 비교 가능한 snapshot이 있고 sample 수 기준도 충족 | 비교 수치 표시 |
| `INSUFFICIENT_DATA` | snapshot은 있으나 sample 수가 기준 미만 | 참고 데이터 부족 안내 |
| `SNAPSHOT_NOT_FOUND` | 조건에 맞는 snapshot이 아직 없음 | 관리자 수집/집계 필요 안내 |
| `UNIT_LAWD_CD_MISSING` | unit의 `lawdCd`가 없음 | 주소 정규화 필요 안내 |
| `UNIT_AREA_MISSING` | unit의 전용면적 값이 없음 | 비교 불가 안내 |

현재 code enum에는 `NO_UNIT`, `NO_LAWD_CODE`, `NO_SNAPSHOT`, `LOW_SAMPLE`, `EMPTY`가 없다.
옛 문서나 초안에 해당 표현이 있으면 현재 API 기준으로 읽지 않는다.

## 11. 남은 리스크

| 리스크 | 설명 | 후속 방향 |
| - | - | - |
| 공공 법정동 전체 seed 실제 적재 | API는 있지만 전체 원본 파일은 repo에 없다. | 최신 공공 파일을 운영 DB에 import |
| 괄호 법정동 없는 도로명주소 | `인천광역시 남동구 은봉로 297`처럼 법정동 token이 없으면 현재 확정 불가 | Juso API 또는 geocoding 기반 보강 |
| scheduler 정책 | batch API는 있지만 자동 실행 주기는 없다. | 운영 주기/대상 지역/월 정책 확정 후 scheduler 연결 |
| 여러 월/지역 일괄 처리 | batch API는 요청 1건 단위다. | 호출자 순회 또는 multi-job batch 설계 |
| 비교 정책 고도화 | 현재는 면적 범위 snapshot 수동 지정 방식 | sourceType 자동 선택, 최근 N개월 default, 면적 허용 오차 설계 |
