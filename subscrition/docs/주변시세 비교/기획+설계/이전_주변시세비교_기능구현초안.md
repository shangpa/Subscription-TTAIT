# [이전] 주변 시세 비교 기능 구현 초안

<!-- markdownlint-disable MD013 MD060 -->

## 1. 문서 목적

공고 상세 화면에서 사용자가 `주변 시세 비교` 기능을 실행했을 때, 해당 공고 주변의 전월세 실거래가를 지도와 리스트로 보여주는 기능의 1차 구현 방향을 정리한다.

이 문서는 바로 구현하기 위한 최종 명세가 아니라, 현재 코드베이스와 데모 구현을 확인한 뒤 정리한 설계 초안이다.

## 2. 현재 확인된 코드 기준

| 구분 | 확인 내용 |
|-|-|
| 정식 백엔드 | `subscrition`에는 공고, 상세, 단위 공급 row 저장 구조가 있음 |
| 정식 프론트 | `app`에는 아직 지도 라이브러리 의존성이 없음 |
| 데모 지도 | `before/frontend`, `before/Design System` 쪽은 `Leaflet + OpenStreetMap tile` 기반 |
| 데모 시세 비교 | `before/demo/demo`에 `MarketRentSnapshot`, `LawdCodeMapping`, `MarketComparisonService`가 있음 |
| 좌표 데이터 | 1차 MVP에서 `AnnouncementUnit.latitude`, `AnnouncementUnit.longitude`, `geocodeStatus`, `geocodedAt` 저장 구조 추가 완료 |
| 법정동 코드 | `Announcement.legalCode` 필드는 있으나 실제 수집 흐름에서 안정적으로 채워지는 구조는 아직 부족함 |

## 3. 데모 구현에서 참고할 부분

### 3.1 지도

참고 파일:

- `before/frontend/src/components/common/MiniMap.tsx`
- `before/Design System/ui_kits/web/MapView.jsx`

확인 내용:

- `Leaflet` 사용
- `OpenStreetMap` tile 사용
- 단일 위치 미니맵과 다중 마커 지도 프로토타입이 있음
- 다중 마커 지도는 마커 클릭 시 요약 카드가 열리고 상세보기로 이동하는 구조

주의할 점:

- 현재 정식 `app`에는 지도 라이브러리가 없음
- 다른 팀원이 공고 마커 지도 기능을 작업 중이므로, 최종 지도 라이브러리는 그 작업물과 맞춰야 함
- 데모 코드를 그대로 복사하기보다 지도 컴포넌트 계약만 참고하는 것이 안전함

### 3.2 시세 비교

참고 파일:

- `before/demo/demo/src/main/java/com/example/demo/market/service/MarketComparisonService.java`
- `before/demo/demo/src/main/java/com/example/demo/market/domain/MarketRentSnapshot.java`
- `before/demo/demo/src/main/java/com/example/demo/market/domain/LawdCodeMapping.java`
- `before/demo/demo/src/main/java/com/example/demo/announcement/dto/MarketComparisonResponse.java`

데모 로직 요약:

1. 공고 주소에서 시도/시군구를 추출한다.
2. `LawdCodeMapping`으로 법정동/시군구 코드를 찾는다.
3. `MarketRentSnapshot`에서 주택 유형과 전용면적 범위가 유사한 실거래가를 조회한다.
4. 평균 보증금과 평균 월세를 계산한다.
5. 공고의 보증금/월세와 비교한 요약 문구를 만든다.

한계:

- 실제 좌표 기반 반경 검색은 아님
- 지도에 실거래가 마커를 찍는 기능은 없음
- 행정구역과 면적 기준 평균 비교에 가까움

## 4. 목표 기능 정의

사용자 관점의 목표:

1. 공고 상세 화면에서 `주변 시세 비교하기` 버튼을 누른다.
2. 해당 공고 또는 선택한 공급 단위의 위치를 기준으로 주변 실거래가를 조회한다.
3. 지도에는 기준 공고와 주변 실거래가가 마커로 표시된다.
4. 리스트에는 주변 실거래가의 건물명, 주소, 전용면적, 보증금, 월세, 계약월이 표시된다.
5. 상단 요약 카드에는 평균 시세 대비 공고의 보증금/월세가 얼마나 낮거나 높은지 표시된다.

## 5. 권장 구현 방향

### 5.1 전체 방향

처음부터 지도 완성형으로 들어가지 말고 다음 순서로 진행한다.

1. 백엔드 시세 비교 API 계약 확정
2. 정식 백엔드에 시세 snapshot 모델 추가
3. 공급 단위의 좌표 저장 구조는 1차 MVP에서 완료했으므로, 다음 단계는 주소 정밀도와 법정동 코드 보강
4. 공고 상세 화면에 요약 카드와 리스트 먼저 연결
5. 팀원 지도 기능과 라이브러리/마커 DTO를 맞춘 뒤 지도 표시 추가

### 5.2 왜 백엔드 우선인가

실거래가 API key, 지오코딩 API key, 데이터 정규화 로직은 모두 서버에 있어야 한다.

프론트에서 외부 API를 직접 호출하면 다음 문제가 생긴다.

- API key 노출
- 호출 제한과 장애 처리 어려움
- XML 파싱과 주소 정규화 중복
- 사용자 클릭마다 외부 API를 호출해 응답 속도와 안정성 저하

따라서 프론트는 정리된 DTO를 받아 지도와 리스트만 렌더링한다.

## 6. 백엔드 설계 초안

### 6.1 위치 데이터 보강

현재 정식 백엔드는 1차 MVP에서 `AnnouncementUnit` 기준 좌표와 Geocoding 상태 저장 구조를 확보했다.
다만 법정동 코드와 주소 정밀도는 아직 부족하다.

주변 비교를 위해 아래 중 하나를 선택해야 한다.

| 방식 | 설명 | 장단점 |
|-|-|-|
| 공고 단위 좌표 | `Announcement`에 좌표 저장 | 구현 단순, 다단지 공고에는 부정확할 수 있음 |
| 공급 단위 좌표 | `AnnouncementUnit`에 좌표 저장 | 1차 MVP 완료. 지도/시세 비교 기준으로 사용 |
| 혼합 방식 | parent에는 대표 좌표, unit에는 개별 좌표 | 후속 확장 후보. parent 대표 좌표가 필요한 화면에서 검토 |

권장 필드:

```text
latitude
longitude
geocode_status
geocoded_at
```

좌표 저장 대상:

- 1차 MVP 완료: `AnnouncementUnit`
- 후속 확장: 필요 시 `Announcement` parent 대표 좌표

### 6.2 시세 데이터 모델

데모의 `MarketRentSnapshot`, `LawdCodeMapping` 구조를 정식 패키지에 맞게 재설계한다.

예상 패키지:

```text
com.ttait.subscription.market
```

예상 엔티티:

```text
MarketRentSnapshot
LawdCodeMapping
```

`MarketRentSnapshot` 주요 필드:

```text
sourceType              // APT_RENT, ROW_HOUSE_RENT, OFFICETEL_RENT 등
lawdCode
dealYm
buildingName
legalDongName
roadName
jibun
buildYear
exclusiveArea
floor
depositAmount
monthlyRentAmount
contractType
latitude
longitude
```

`LawdCodeMapping` 주요 필드:

```text
sidoName
sigunguName
sigunguCode5
fullCode10
active
```

### 6.3 외부 API 연동

후보 API:

- 국토교통부 아파트 전월세 실거래가
- 국토교통부 연립다세대 전월세 실거래가
- 필요 시 오피스텔 전월세 실거래가
- 주소 좌표 변환용 geocoding API

주의:

- 실거래가 API는 보통 `LAWD_CD`, `DEAL_YMD` 중심이다.
- 응답은 XML 기반으로 들어오는 경우가 많다.
- “반경 n m” 검색은 외부 API가 바로 해주는 것이 아니라, 서버가 좌표를 저장한 뒤 직접 필터링해야 한다.

### 6.4 API 초안

공고 상세와 분리된 별도 API를 권장한다.

```http
GET /api/announcements/{announcementId}/market-comparison?unitId={unitId}&radiusMeters=1000&months=6&limit=30
```

분리하는 이유:

- 공고 상세 로딩과 시세 비교 로딩을 분리 가능
- 사용자가 버튼을 눌렀을 때만 호출 가능
- 외부/캐시 데이터가 부족해도 공고 상세 조회는 영향 없음

응답 예시:

```json
{
  "status": "OK",
  "target": {
    "announcementId": 1,
    "unitId": 10,
    "address": "경기도 오산시 ...",
    "latitude": 37.152,
    "longitude": 127.077,
    "depositAmount": 1119,
    "monthlyRentAmount": 15,
    "exclusiveArea": 46.5
  },
  "summary": {
    "sampleCount": 18,
    "averageDepositAmount": 2800,
    "averageMonthlyRentAmount": 32,
    "depositSavingRate": 60,
    "rentSavingRate": 53,
    "message": "주변 시세 대비 월세 부담이 낮습니다."
  },
  "items": [
    {
      "id": 1,
      "buildingName": "은마",
      "address": "서울 강남구 대치동 ...",
      "latitude": 37.49,
      "longitude": 127.06,
      "exclusiveArea": 76.79,
      "depositAmount": 70000,
      "monthlyRentAmount": 0,
      "dealYm": "202403"
    }
  ]
}
```

상태값 후보:

| status | 의미 |
|-|-|
| `OK` | 비교 가능 |
| `NO_COORDINATE` | 공고 또는 단위 좌표 없음 |
| `INSUFFICIENT_DATA` | 비교 가능한 실거래 데이터 부족 |
| `UNSUPPORTED_HOUSE_TYPE` | 지원하지 않는 주택 유형 |
| `EXTERNAL_DATA_UNAVAILABLE` | 외부 데이터 수집 실패 또는 미수집 |

```text
MarketComparisonPanel
MarketSummaryCard
```

### 7.3 지도 연동 전략

팀원이 작업 중인 공고 마커 지도와 충돌하지 않도록 지도 라이브러리를 직접 물고 들어가지 않는다.

공용 마커 DTO를 먼저 정의한다.

```js
{
  id,
  type: 'target' | 'market' | 'announcement',
  lat,
  lng,
  title,
  priceLabel,
  metaLabel
}
```

지도 라이브러리가 확정되면 `MapAdapter` 내부에서만 `Leaflet`, `Kakao Map`, `Naver Map` 중 하나를 사용한다.

## 8. 구현 단계 제안

### 1단계: 결정 사항 확정

- 팀원 지도 기능의 라이브러리 확인
- 기준 위치를 공고 parent로 할지, unit으로 할지 결정
- 좌표 변환 API 선택
- 지원할 실거래가 유형 선택: 아파트, 연립다세대, 오피스텔 등
- 기본 반경과 조회 기간 결정: 예) 1km, 최근 6개월

### 2단계: 백엔드 기반 작업

- `market` 패키지 추가
- `MarketRentSnapshot`, `LawdCodeMapping` 추가
- 실거래가 XML 파싱 adapter 추가
- 좌표/Geocoding 저장 구조는 1차 MVP 완료. 다음 작업은 법정동 코드, 주소 정밀도, RTMS 연계
- 비교 service 테스트 작성

### 3단계: API 작업

- `GET /api/announcements/{announcementId}/market-comparison` 추가
- 데이터 부족/좌표 없음/미지원 주택유형 상태 응답 처리
- public DTO에 admin-only 원문/raw/source key가 섞이지 않도록 분리

### 4단계: 프론트 UI 작업

- 공고 상세에 버튼 추가
- API 호출 상태 관리
- 요약 카드와 리스트 먼저 구현
- 지도는 팀원 지도 컴포넌트와 계약 확정 후 연결

### 5단계: 지도 연결

- 기준 공고 마커 표시
- 주변 실거래가 마커 표시
- 리스트 항목 hover/click 시 지도 마커 강조
- 지도 마커 클릭 시 리스트 항목 또는 상세 카드 표시

## 9. 주요 리스크

| 리스크 | 설명 | 대응 |
|-|-|-|
| 좌표 없음 | 일부 공고는 `units=[]`이거나 주소가 부정확해 좌표 기준 비교가 어려움 | 단위 row 미추출 정책과 주소 정규화 보강 |
| 법정동 코드 미완성 | `legalCode`가 안정적으로 채워지지 않음 | 법정동 매핑 테이블과 주소 정규화 필요 |
| 실거래가 데이터 부족 | 공고 위치/면적/주택유형에 맞는 샘플이 적을 수 있음 | status를 `INSUFFICIENT_DATA`로 분리 |
| 지도 라이브러리 충돌 | 팀원 지도 기능과 중복 구현 가능 | `MapAdapter`와 공용 marker DTO 사용 |
| 외부 API 호출 제한 | 사용자가 클릭할 때마다 호출하면 느리고 불안정 | snapshot/cache 기반으로 조회 |
| public DTO 노출 범위 | admin-only raw/source 필드가 섞일 수 있음 | public/admin DTO 분리 유지 |

## 10. 1차 MVP 범위 제안

주변시세 비교 기능 본 구현의 MVP에서는 아래까지만 목표로 잡는다.
좌표/Geocoding 기반 1차 MVP는 별도 선행 작업으로 완료된 상태다.

- 공고 상세에서 `주변 시세 비교하기` 버튼 제공
- 백엔드 비교 API 제공
- 주변 실거래가 리스트 제공
- 평균 보증금/월세 비교 요약 제공
- 좌표가 있는 경우에만 지도 표시

1차 MVP에서 제외해도 되는 것:

- 지도 클러스터링
- 실시간 외부 API 호출
- 모든 주택 유형 지원
- 지도에서 공고 상세 이동까지 포함한 통합 지도 기능
- AI 당첨 가능성 리포트와의 결합

## 11. 최종 권장 방향

현재 상태에서는 지도부터 구현하지 말고, 백엔드 시세 비교 API와 데이터 모델을 먼저 확정하는 것이 안전하다.

특히 “주변”이라는 요구사항은 좌표가 있어야 제대로 구현할 수 있으므로, 좌표 저장과 geocoding 정책이 선행되어야 한다.

프론트는 팀원이 만드는 공고 마커 지도와 같은 지도 라이브러리를 사용하되, 직접 결합하지 않고 `MapAdapter`와 공용 marker DTO로 연결하는 구조가 가장 안전하다.
