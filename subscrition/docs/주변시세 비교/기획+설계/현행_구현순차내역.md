# [현행] 주변시세 비교 구현 순차 내역

<!-- markdownlint-disable MD013 -->

## 1. 현재 1차 MVP 범위

현재 branch의 완료 기준은 주변시세 비교 전체가 아니라 좌표와 Geocoding 기반 구축이다.
아래 범위만 현재 MVP deliverable로 본다.

| 항목 | 현재 결정 | 현재 branch 범위 |
| - | - | - |
| 기준 단위 | `AnnouncementUnit` | 포함 |
| 좌표 변환 | `Naver Geocoding` | 포함 |
| 좌표 매핑 | `x=longitude`, `y=latitude` | 포함 |
| 좌표 저장 | `AnnouncementUnit.latitude`, `AnnouncementUnit.longitude` | 포함 |
| 상태 저장 | `GeocodeStatus`, `geocodedAt` | 포함 |
| import 연결 | 신규 LH import 후 Geocoding 보강 | 포함 |
| 조회 응답 | public/admin unit DTO 좌표와 status 노출 | 포함 |
| 테스트 | service, client, DTO, import orchestration 검증 | 포함 |

`AnnouncementUnit`이 좌표의 기준이다.
Naver Geocoding 응답에서는 `x`가 경도, `y`가 위도이므로 저장 시 `x=longitude`, `y=latitude`로 매핑한다.

## 2. 현재 branch에서 제외하는 범위

아래 항목은 현재 branch 구현 범위가 아니다.
후속 작업이나 참고 맥락으로만 본다.

| 항목 | 현재 판단 | 메모 |
| - | - | - |
| Dynamic Map 화면 | 제외 | 지도 렌더링과 marker 이동은 후속 UI 작업 |
| 프론트 지도 연결 | 제외 | `MapAdapter`, marker DTO는 후속 설계 |
| RTMS 실거래가 수집 | 제외 | 국토부 API, XML parsing, 저장 정책 필요 |
| 법정동 코드, `LAWD_CD` | 제외 | Geocoding만으로 자동 해결되지 않음 |
| `market snapshot` 저장 | 제외 | 주변시세 데이터 저장 구조는 후속 작업 |
| 주변시세 비교 로직 | 제외 | 반경, 기간, 면적 보정, 평균/중앙값 정책 필요 |

이 제외 항목은 기획상 중요하지만, 현재 MVP의 완료 여부를 판단하는 기준으로 쓰지 않는다.

## 3. 1차 MVP 구현 순서

현재 branch는 아래 순서로 좌표와 Geocoding 기반을 먼저 확보한다.

| 순서 | 작업 | 이유 |
| - | - | - |
| 1 | 기준 위치를 `AnnouncementUnit`으로 확정 | 실제 신청 가능한 공급 row 기준으로 좌표를 저장하기 위함 |
| 2 | 좌표와 Geocoding 상태 필드 추가 | `latitude`, `longitude`, `geocodeStatus`, `geocodedAt` 저장 |
| 3 | Naver Geocoding client 결과를 typed result로 변환 | raw JSON 의존을 줄이고 좌표 매핑을 테스트하기 위함 |
| 4 | 신규 import 후 Geocoding 보강 연결 | import 성공과 외부 API 실패를 분리하기 위함 |
| 5 | public/admin unit DTO에 좌표와 status 노출 | 조회 응답에서 좌표 유무와 상태를 확인하기 위함 |
| 6 | tests-after 검증 | 좌표 매핑, 실패 처리, DTO 노출, import orchestration 확인 |

## 4. 후속 순서

아래 순서는 현재 branch 완료 이후의 backlog다.
현재 MVP 검증이나 final review의 scope 판단에 포함하지 않는다.

| 순서 | 후속 작업 | 선행 조건 |
| - | - | - |
| 1 | 법정동 코드와 주소 정규화 | RTMS 조회 기준 확정 |
| 2 | RTMS 수집과 XML parsing | 외부 API 계약과 저장 정책 확정 |
| 3 | `market snapshot` 저장 구조 | cache, 갱신 주기, 기준 날짜 확정 |
| 4 | 주변시세 비교 API | 반경, 기간, 면적 보정, 통계 정책 확정 |
| 5 | Dynamic Map과 marker 표시 | 백엔드 응답 계약과 프론트 화면 확정 |
| 6 | 공고 상세의 지도, 리스트, 요약 연결 | 비교 API와 UI 상태 설계 확정 |

## 5. 기존 기획과의 관계

기존 `현행_주변시세비교_기능구현초안.md`는 주변시세 비교 전체 방향을 다룬다.
이 문서는 그중 현재 branch에서 실제로 끝낼 1차 MVP 범위를 좁혀 적는다.

따라서 RTMS, 법정동 코드, Dynamic Map, 비교 계산은 삭제된 요구사항이 아니다.
현재 branch의 완료 기준에서 제외하고 후속 backlog로 분리한 것이다.

## 6. 결론

현재 branch는 `AnnouncementUnit` 좌표 저장과 Naver Geocoding 검증을 끝내는 단계다.
Dynamic Map, RTMS, `market snapshot`, 주변시세 비교 계산은 후속 범위로만 다룬다.
