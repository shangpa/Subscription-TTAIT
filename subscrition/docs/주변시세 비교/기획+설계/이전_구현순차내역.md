# [이전] 주변시세 비교 구현 순차 내역

<!-- markdownlint-disable MD013 -->

## 1. 현재 1차 MVP 범위

현재 branch의 완료 기준은 좌표와 Geocoding 기반 구축, RTMS raw 수집, snapshot 집계, public 주변시세 비교 API, 법정동 seed import, RTMS/snapshot batch API까지다.
프론트 화면 구현과 scheduler형 완전 자동 운영은 현재 deliverable에서 제외한다.

| 항목 | 현재 결정 | 현재 branch 범위 |
| - | - | - |
| 기준 단위 | `AnnouncementUnit` | 포함 |
| 좌표 변환 | `Naver Geocoding` | 포함 |
| 좌표 매핑 | `x=longitude`, `y=latitude` | 포함 |
| 좌표 저장 | `AnnouncementUnit.latitude`, `AnnouncementUnit.longitude` | 포함 |
| 상태 저장 | `GeocodeStatus`, `geocodedAt` | 포함 |
| import 연결 | 신규 LH import 후 Geocoding 보강 | 포함 |
| 조회 응답 | public/admin unit DTO 좌표와 status 노출 | 포함 |
| 테스트 | service, client, DTO, import orchestration 검증 | 포함 |
| Apidog 수동 검증 | legacy import 후 admin review detail에서 `SUCCESS` 좌표 확인 | 포함 |
| RTMS 수동 수집 | 관리자 API로 단일 page 수집 후 raw transaction 저장 | 포함 |
| `MarketPriceSnapshot` 집계 | 관리자 API로 raw transaction을 평균/중앙값 snapshot으로 저장 | 포함 |
| 주변시세 비교 API | public GET API로 `AnnouncementUnit` 가격과 snapshot 평균 비교 | 포함 |
| 프론트 인수인계 문서 | 팀원 AI가 읽을 API 계약/status/실패 케이스 정리 | 포함 |
| 법정동 seed import | 공공 법정동 코드 CSV/TSV 원문 일괄 적재 | 포함 |
| RTMS/snapshot batch | RTMS 전체 수집 후 snapshot 집계 연결 | 포함 |
| Swagger 예시 | 주변시세 관련 요청/응답 예시 보강 | 포함 |

`AnnouncementUnit`이 좌표의 기준이다.
Naver Geocoding 응답에서는 `x`가 경도, `y`가 위도이므로 저장 시 `x=longitude`, `y=latitude`로 매핑한다.

2026-05-21 Apidog 수동 테스트에서 `announcementId=6`, `unitId=2` 기준 `geocodeStatus=SUCCESS`, `latitude`, `longitude`, `geocodedAt` 저장과 admin review detail 노출을 확인했다.

## 2. 현재 branch에서 제외하는 범위

아래 항목은 현재 branch 구현 범위가 아니다.
후속 작업이나 참고 맥락으로만 본다.

| 항목 | 현재 판단 | 메모 |
| - | - | - |
| Dynamic Map 화면 | 제외 | 지도 렌더링과 marker 이동은 후속 UI 작업 |
| 프론트 지도 연결 | 제외 | `MapAdapter`, marker DTO는 후속 설계 |
| scheduler형 자동 수집 | 제외 | batch API는 구현 완료, cron 주기와 대상 지역/월 정책은 후속 |
| 도로명주소 법정동 보강 | 일부 제외 | 괄호 법정동은 처리, 괄호 없는 도로명주소는 Juso/geocoding 보강 필요 |
| `market snapshot` 저장 | 포함 | 수동 집계 API 기준 구현 완료 |
| 주변시세 비교 로직 | 포함 | public API 기준 평균값 비교 구현 완료 |

이 제외 항목은 기획상 중요하지만, 현재 MVP의 완료 여부를 판단하는 기준으로 쓰지 않는다.

## 3. 1차 MVP 구현 순서

현재 branch는 아래 순서로 좌표와 Geocoding 기반을 먼저 확보한다.

| 순서 | 작업 | 이유 |
| - | - | - |
| 1 | 기준 위치를 `AnnouncementUnit`으로 확정 | 실제 신청 가능한 공급 row 기준으로 좌표를 저장하기 위함 |
| 2 | 좌표와 Geocoding 상태 필드 추가 | `latitude`, `longitude`, `geocodeStatus`, `geocodedAt` 저장 |
| 3 | Naver Geocoding client 결과를 typed result로 변환 | raw JSON 의존을 줄이고 좌표 매핑을 테스트하기 위함 |
| 4 | 신규 import 후 Geocoding 보강 연결 | import 성공과 외부 API 실패를 분리하기 위함 |
| 5 | public/admin unit DTO에 좌표와 status 노출 | 조회 응답에서 좌표 유무와 상태를 확인하기 위함 |
| 6 | tests-after 검증 | 좌표 매핑, 실패 처리, DTO 노출, import orchestration 확인 |
| 7 | 관리자 RTMS 수동 수집 API 구현 | RTMS client 결과를 raw transaction으로 저장하고 중복 수집을 방지하기 위함 |
| 8 | `MarketPriceSnapshot` 집계 API 구현 | raw 거래를 기간/지역/면적 기준 snapshot으로 저장하기 위함 |
| 9 | 주변시세 비교 API 구현 | 프론트가 공고 unit 기준으로 주변시세 비교 결과를 조회하기 위함 |
| 10 | 인수인계 문서 작성 | 팀원과 팀원 AI가 endpoint/status/실패 케이스를 바로 이해하기 위함 |

## 4. 후속 순서

아래 순서는 현재 branch 완료 이후의 backlog다.
현재 MVP 검증이나 final review의 scope 판단에 포함하지 않는다.

| 순서 | 후속 작업 | 선행 조건 |
| - | - | - |
| 1 | 법정동 코드와 주소 정규화 | RTMS 조회 기준 확정 |
| 2 | `units=[]` 공고 처리 정책 | 단위 row 미추출 공고의 재파싱/수동보완/비교 제외 기준 확정 |
| 3 | 주소 정밀도 개선 | 행정동 수준 주소를 단지/도로명/지번 수준으로 보강 |
| 4 | RTMS 수집 정책 보강 | `resultCode=03`, 전체 pagination, 중복 race 처리 정책 확정 |
| 5 | RTMS/snapshot 자동화 | 현재는 수동 API 기준 완료. batch, scheduler, 재집계 주기 확정 필요 |
| 6 | 비교 정책 고도화 | 면적 허용 오차, 최근 N개월 default, 주택 유형 자동 매핑 확정 필요 |
| 7 | Dynamic Map과 marker 표시 | 백엔드 응답 계약을 바탕으로 프론트 화면 확정 |
| 8 | 공고 상세의 지도, 리스트, 요약 연결 | 비교 API와 UI 상태 설계 확정 |

## 5. 기존 기획과의 관계

기존 `이전_주변시세비교_기능구현초안.md`는 주변시세 비교 전체 방향을 다룬다.
이 문서는 그중 현재 branch에서 실제로 끝낼 1차 MVP 범위를 좁혀 적는다.

따라서 RTMS 후속 정책, 법정동 코드, Dynamic Map, 비교 계산은 삭제된 요구사항이 아니다.
현재 branch의 완료 기준에서 제외하고 후속 backlog로 분리한 것이다.

## 6. 결론

현재 branch는 `AnnouncementUnit` 좌표 저장, Naver Geocoding 검증, 관리자 RTMS 단일 page 수동 수집, raw transaction 저장, `MarketPriceSnapshot` 수동 집계, public 주변시세 비교 API까지 끝낸 단계다.
`units=[]` 공고와 행정동 수준 주소 정밀도는 실패가 아니라 후속 보완 대상으로 분리한다.
Dynamic Map, RTMS 전체 수집 정책, batch 재집계, 비교 정책 고도화는 후속 범위로 다룬다.
