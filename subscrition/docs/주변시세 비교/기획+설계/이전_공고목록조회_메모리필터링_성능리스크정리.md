# [이전] 공고 목록 조회 메모리 필터링 성능 리스크 정리

<!-- markdownlint-disable MD013 -->

## 1. 문서 목적

이 문서는 `AnnouncementQueryService`의 public 공고 목록 조회가 DB pagination이 아니라 전체 조회 후 Java 메모리에서 필터링과 paging을 수행하던 구조를 정리하고, 주변시세 비교 개발 흐름에서 이 문제가 어느 정도 우선순위를 가지는지 판단하기 위한 기준을 남긴다.

2026-05-21 `perf/public-list-pagination` branch에서 public 목록 조회는 custom repository 기반 DB filtering/pagination으로 개선했다. 따라서 아래 2~7장은 개선 전 리스크 분석 기록이고, 8장은 개선 반영 내역이다.

## 2. 결론 요약

현재 구조는 개발 중 즉시 blocker로 보기는 어렵다.

다만 공고 데이터가 수천 건 이상으로 늘거나 public 목록 API가 실제 프론트, 지도, 주변시세 비교 진입점으로 쓰이기 시작하면 병목이 될 가능성이 높다. 따라서 Geocoding 1차 MVP 검증 단계에서는 후순위로 둘 수 있지만, 정식 public 조회/필터링 고도화 전에는 DB query 기반 pagination으로 바꾸는 것이 안전하다.

| 구간 | 위험도 | 판단 |
| - | - | - |
| 로컬/데모, 공고 수십~수백 건 | 낮음~중간 | 전체 조회 비용이 작아 체감 문제가 제한적일 수 있음 |
| LH import 반복, 공고 수천 건 | 높음 | 매 요청마다 전체 public 공고를 로딩하고 service에서 필터링함 |
| 프론트 목록, 지도, 주변시세 비교 연결 | 높음 | 목록 API가 기본 진입점이 되어 응답시간 병목이 될 수 있음 |
| 운영 배포 | 높음 | pagination이 DB가 아니라 JVM 메모리에서 수행되어 데이터 증가에 취약함 |

## 3. 현재 코드 구조

현재 public 공고 목록 조회는 다음 파일에 있다.

```text
subscrition/src/main/java/com/ttait/subscription/announcement/service/AnnouncementQueryService.java
```

핵심 흐름은 다음과 같다.

```java
List<Announcement> announcements = announcementRepository.findPublicVisible(
                PUBLIC_VISIBLE_REVIEW_STATUSES,
                Pageable.unpaged())
        .getContent();

Map<Long, Set<CategoryCode>> categoryMap = loadCategoryMap(extractIds(announcements));
List<Announcement> filtered = announcements.stream()
        .filter(...)
        .sorted(...)
        .toList();

return new PageImpl<>(filtered.subList(start, end), pageable, filtered.size());
```

문제의 핵심은 `Pageable.unpaged()` 자체보다, DB에서 page 단위로 가져오지 않고 전체 public 공고를 가져온 뒤 Java service에서 필터링, 정렬, subList paging을 한다는 점이다.

사용자가 `page=0&size=10`만 요청해도 DB에서는 조건에 맞는 전체 public 공고를 가져오고, 이후 JVM 메모리에서 필요한 10건만 잘라낸다.

## 5. 현재 개발 중 영향도

현재 주변시세 비교 작업의 초점은 `AnnouncementUnit` 좌표 저장과 Naver Geocoding 기반 구축이다. 이 범위에서는 public 목록 API 성능 문제가 직접적인 blocker는 아니다.

그러나 아래 작업으로 넘어가기 전에는 수정 우선순위를 올리는 것이 좋다.

1. public 공고 목록을 정식 프론트 화면에 연결한다.
2. LH import를 반복해서 공고 데이터를 많이 쌓는다.
3. 지도, 마커, 주변시세 비교 진입점을 공고 목록과 연결한다.
4. 운영 또는 장시간 시연 환경에 배포한다.

특히 주변시세 비교 기능이 공고 상세 진입, 지도 표시, 필터링과 결합되면 목록 API는 사용자가 가장 자주 호출하는 경로가 될 가능성이 높다. 이때 전체 조회 방식은 응답시간과 메모리 사용량 양쪽에서 병목이 될 수 있다.

## 6. 권장 수정 방향

최종 방향은 필터링과 pagination을 DB query로 내리는 것이다.

권장 우선순위는 다음과 같다.

| 우선순위 | 수정 범위 | 설명 |
| - | - | - |
| 1 | public visible 조건 | `APPROVED`, `CORRECTED`, `deleted=false`, `merged=false`를 DB query에 유지 |
| 2 | 기본 필터 | 지역, 공급유형, 주택유형, provider, status, keyword, 가격 범위를 DB 조건으로 이동 |
| 3 | DB pagination | repository query에 `Pageable`을 그대로 전달하고 DB에서 page 단위 조회 |
| 4 | category 필터 | join/subquery 또는 2단계 조회로 설계. 단, 전체 테이블 로딩은 피함 |
| 5 | 정렬 | `applicationEndDate`, `id` 정렬을 DB `ORDER BY`로 이동 |

QueryDSL을 바로 도입하지 않더라도 Spring Data JPA `Specification`, `@Query`, 또는 단계적 repository method 추가로 1차 개선이 가능하다.

## 7. 최종 판단

현재 Geocoding 1차 MVP 검증 단계에서는 이 문제를 즉시 수정하지 않아도 된다.

다만 이 구조는 데이터가 적을 때만 괜찮은 임시 구현에 가깝다. 주변시세 비교 본기능, 지도, public 프론트 연결 전에 `AnnouncementQueryService`의 목록 조회는 DB query 기반 pagination으로 개선하는 것을 후속 backlog에 올리는 것이 맞다.

## 8. 개선 반영 내역

`perf/public-list-pagination` branch에서 public 공고 목록 조회의 기본 필터와 pagination을 DB query로 내렸다.

변경 후 핵심 흐름은 다음과 같다.

```java
AnnouncementSearchCondition condition = new AnnouncementSearchCondition(...);

return announcementRepository.searchPublicVisible(condition, pageable)
        .map(this::toListItem);
```

주요 반영 범위는 다음과 같다.

| 항목 | 반영 내용 |
| - | - |
| public visible 조건 | `APPROVED`, `CORRECTED`, `deleted=false`, `merged=false`를 DB query 조건으로 유지 |
| 기본 필터 | 지역, 공급유형, 주택유형, provider, status, keyword, 보증금/월세 범위를 DB 조건으로 이동 |
| category 필터 | `announcement_category`에 저장된 category를 우선하고, 저장 category가 없는 공고만 기존 키워드 fallback 사용 |
| pagination | `Pageable` offset/size를 DB query에 적용하고 count query로 `totalElements` 계산 |
| 정렬 | `applicationEndDate` 오름차순, null last, `id` 내림차순을 DB query에 적용 |

검증은 `AnnouncementSearchRepositoryTest`, 전체 test, build로 확인했다.
