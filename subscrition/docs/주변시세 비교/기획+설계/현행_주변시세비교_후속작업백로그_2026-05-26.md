### 3.2 확인 순서

1. LH 수집 API를 실제로 호출한다.
2. 수집 결과로 `announcement`, `announcement_unit`이 생성되는지 확인한다.
3. 대상 unit에 `exclusive_area_value`, `deposit_amount`, `monthly_rent_amount`가 있는지 확인한다.
4. 대상 unit에 `lawd_cd`가 채워져 있는지 확인한다.
5. `lawd_cd`가 없으면 주소 정규화/법정동 매핑 후속 작업이 필요하다.
6. RTMS 수동 수집 API로 같은 `lawdCd`, `dealYm`의 raw 거래를 수집한다.
7. `market_transaction_raw`에 row가 저장되는지 확인한다.
8. snapshot 집계 API를 호출한다.
9. `market_price_snapshot`에 snapshot이 저장되고 `status`가 `OK` 또는 `INSUFFICIENT_DATA`인지 확인한다.
10. 주변시세 비교 public API를 실제 `announcementId`, `unitId`로 호출한다.
11. `COMPARABLE`, `SNAPSHOT_NOT_FOUND`, `UNIT_LAWD_CD_MISSING`, `UNIT_AREA_MISSING`, `INSUFFICIENT_DATA` 중 어떤 status가 나오는지 기록한다.

## 4. 후속 작업 우선순위

| 우선순위 | 작업 | 이유 |
| - | - | - |
| 1 | `lawdCodeMapping` 원천 데이터 확보/대량 적재 자동화 | 관리자 JSON upsert API는 추가됐지만 공공 법정동 코드 전체 seed는 아직 없음 |
| 2 | 주소 정규화 운영 검증 | 매핑 적재 후 `normalize-units` 재처리 API로 실제 LH unit `lawdCd` 자동 저장 확인 필요 |
| 3 | RTMS `resultCode=03 No Data` 정책 확정 | 현재 `resultCode != 00`은 실패로 처리하므로 빈 데이터 정책 정리 필요 |
| 4 | RTMS 전체 pagination 수집 | 현재는 단일 page 수동 수집만 완료 |
| 5 | snapshot 자동 재집계 batch/scheduler | 운영에서 매번 수동 집계하기 어렵기 때문 |
| 6 | 비교 기준 고도화 | 면적 허용 오차, 최근 N개월 default, sourceType 자동 매핑 필요 |
| 7 | 프론트 상태 표시 정책 확정 | status별 UI 문구와 fallback을 일관되게 맞추기 위함 |
| 8 | 운영 로그/재시도 정책 | 외부 API 실패와 수집 실패를 추적하기 위함 |

## 5. 팀원에게 넘길 때 같이 전달할 내용

- public 비교 API는 이미 생성된 snapshot을 조회하는 API다.
- 프론트가 직접 RTMS를 호출하지 않는다.
- `SNAPSHOT_NOT_FOUND`는 프론트 오류가 아니라 해당 조건의 snapshot이 아직 없다는 뜻이다.
- `UNIT_LAWD_CD_MISSING`은 주소 정규화/법정동 코드 매핑이 선행되지 않았다는 뜻이다.
- `INSUFFICIENT_DATA`는 snapshot은 있으나 sample 수가 부족하다는 뜻이다.
- 실제 UI에서는 `COMPARABLE`만 수치 비교를 강조하고, 나머지는 안내 문구로 처리하는 편이 안전하다.

## 7. `lawdCd` 자동 정규화 진행 상황

2026-05-26 후속 작업에서 LH import/reimport 후 `AnnouncementUnit` 주소 정규화가 자동 실행되도록 연결했다.

현재 동작:

```text
LH detail import/reimport
→ announcement_unit 교체 저장
→ AddressNormalizationService 자동 호출
→ lawd_code_mapping에 매핑이 있으면 legal_dong_code/lawd_cd 저장
→ 이후 Naver geocoding 후처리 실행
```

주의:

- 로컬 DB 기준 `lawd_code_mapping` 데이터가 0건이면 자동 호출은 되지만 unit은 `NO_LAWD_CODE` 상태가 된다.
- 실운영에서 `COMPARABLE`까지 자동 연결하려면 법정동 매핑 seed 적재 또는 관리자 재처리 API가 필요하다.
- 도/광역시가 포함된 주소(`경기도 김포시 마산동 ...`, `인천광역시 부평구 부평동 ...`)는 `시/군/구` 다음 token을 법정동 후보로 해석한다.

## 8. 법정동 매핑 관리자 API 진행 상황

2026-05-26 후속 작업에서 다음 관리자 API를 추가했다.

| 용도 | Endpoint | 메모 |
| - | - | - |
| 법정동 매핑 bulk upsert | `POST /api/admin/market/address/lawd-code-mappings` | JSON으로 10자리 `legalDongCode` 적재 |
| 공고 unit 주소 정규화 재처리 | `POST /api/admin/market/address/announcements/{announcementId}/normalize-units` | `retryNoLawdCode=true`로 기존 실패 건 재처리 |

남은 부분은 공공 법정동 코드 전체 데이터를 어떤 방식으로 가져와 seed할지 결정하는 것이다.
