### 3.2 확인 순서

1. LH 수집 API를 실제로 호출한다.
2. 수집 결과로 `announcement`, `announcement_unit`이 생성되는지 확인한다.
3. 대상 unit에 `exclusive_area_value`, `deposit_amount`, `monthly_rent_amount`가 있는지 확인한다.
4. 대상 unit에 `lawd_cd`가 채워져 있는지 확인한다.
5. `lawd_cd`가 없으면 주소 정규화/법정동 매핑 후속 작업이 필요하다.
6. RTMS 수동 수집 API로 같은 `lawdCd`, `dealYm`의 raw 거래를 수집한다.
7. `market_transaction_raw`에 row가 저장되는지 확인한다.
8. snapshot 집계 API를 호출한다.
9. `market_price_snapshot`에 snapshot이 저장되고 `status`가 `OK` 또는 `INSUFFICIENT_DATA`인지 확인한다.
10. 주변시세 비교 public API를 실제 `announcementId`, `unitId`로 호출한다.
11. `COMPARABLE`, `SNAPSHOT_NOT_FOUND`, `UNIT_LAWD_CD_MISSING`, `UNIT_AREA_MISSING`, `INSUFFICIENT_DATA` 중 어떤 status가 나오는지 기록한다.

## 4. 후속 작업 우선순위

현재 main 기준 `법정동 seed import`, `normalize-units`, `RTMS collect-all`, `RTMS/snapshot batch`, `public comparison API`는 구현 완료다.
아래는 남은 운영/고도화 backlog다.

| 우선순위 | 작업 | 이유 |
| - | - | - |
| 1 | 최신 공공 법정동 코드 전체 파일 운영 적재 | endpoint는 준비됐지만 실제 전체 원본 파일은 repo에 포함하지 않음 |
| 2 | 도로명주소 법정동 보강 | 괄호 법정동이 없는 도로명주소는 현재 법정동을 확정하지 못할 수 있음 |
| 3 | scheduler 정책 결정 | batch API는 준비됐고, 자동 실행 주기/대상 지역/월 정책이 필요 |
| 4 | multi-month/multi-region batch 편의 API | 현재 batch는 요청 1건 기준이므로 여러 월/지역 반복은 호출자가 순회해야 함 |
| 5 | 비교 기준 고도화 | 면적 허용 오차, 최근 N개월 default, sourceType 자동 매핑 필요 |
| 6 | 프론트 status별 UI copy | `COMPARABLE`, `SNAPSHOT_NOT_FOUND`, `INSUFFICIENT_DATA`, `UNIT_LAWD_CD_MISSING`, `UNIT_AREA_MISSING` 표시 정책 필요 |
| 7 | 운영 로그/재시도 정책 | 외부 API 실패와 수집 실패를 추적하기 위함 |

## 5. 팀원에게 넘길 때 같이 전달할 내용

- 프론트가 직접 RTMS를 호출하지 않는다.
- public 비교 API는 이미 생성된 snapshot을 조회하는 API다.
- `SNAPSHOT_NOT_FOUND`는 프론트 오류가 아니라 해당 조건의 snapshot이 아직 없다는 뜻이다.
- `UNIT_LAWD_CD_MISSING`은 주소 정규화/법정동 코드 매핑이 선행되지 않았다는 뜻이다.
- `UNIT_AREA_MISSING`은 LH/PDF unit row에서 전용면적 값이 없다는 뜻이다.
- `INSUFFICIENT_DATA`는 snapshot은 있으나 sample 수가 부족하다는 뜻이다.
- 실제 UI에서는 `COMPARABLE`만 수치 비교를 강조하고, 나머지는 안내 문구로 처리하는 편이 안전하다.
