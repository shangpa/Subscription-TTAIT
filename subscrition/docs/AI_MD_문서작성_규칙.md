# AI Markdown 문서 작성 규칙

이 문서는 팀원이 AI를 사용해 Markdown 문서를 만들거나 수정하기 전에 확인해야 하는 공통 규칙이다.

## 1. 문서 상태를 먼저 정한다

새 문서를 만들거나 기존 문서를 바꾸기 전에 문서 상태를 먼저 정한다.

- `현행`: 현재 프로젝트 기준으로 참고해야 하는 문서
- `이전`: 과거 작업 기록, 초기 기획, 중간발표 이전 자료, 대체된 문서

상태가 애매하면 임의로 정하지 말고 팀원에게 먼저 확인한다.

## 2. 파일명 규칙

파일명만 봐도 문서 상태와 내용을 알 수 있게 작성한다.

```text
현행_문서주제_상세내용.md
이전_문서주제_상세내용.md
```

예시:

```text
현행_공고조회_API가이드.md
현행_LH공고수집_PDF_AI파싱_기능명세.md
이전_AI파싱_품질문제분석_v1.md
이전_프로젝트분석_초기문서.md
```

규칙:

- 파일명에는 공백 대신 `_`를 사용한다.
- `API`, `AI`, `PDF`, `DB`, `LH`, `OpenAI` 같은 기술 약어는 유지한다.
- 실제 버전 이력이 있는 문서만 `v1`, `v2`를 붙인다.
- 폴더명은 특별한 이유 없이 바꾸지 않는다.

## 3. 제목(H1) 규칙

문서의 첫 번째 H1 제목에도 상태를 표시한다.

```markdown
# [현행] 공고 조회 API 가이드
# [이전] AI 파싱 품질 문제 분석 v1
```

규칙:

- 문서에는 첫 번째 H1이 반드시 있어야 한다.
- 상태 표기는 첫 번째 H1에만 붙인다.
- 코드블록 안의 `# 주석`은 제목으로 바꾸면 안 된다.
- 본문 중간 제목에는 불필요하게 `[현행]`, `[이전]`을 반복하지 않는다.

## 4. AI에게 시킬 때 반드시 포함할 조건

AI에게 Markdown 문서 작성이나 수정 작업을 맡길 때 아래 조건을 명시한다.

```text
- 파일명은 현행_/이전_ prefix를 사용한다.
- 첫 번째 H1은 # [현행] ... 또는 # [이전] ... 형식으로 작성한다.
- 본문 내용은 요청한 범위 외에는 임의로 재작성하지 않는다.
- 기존 문서를 rename할 경우 내부 참조와 다른 문서의 링크도 함께 확인한다.
- 코드블록, 명령어 예시, 주석은 제목 변경 대상으로 보지 않는다.
```

## 5. 기존 문서를 수정할 때 검증한다

파일명을 바꾸거나 제목을 바꾼 뒤에는 반드시 기존 파일명 참조가 남아 있는지 확인한다.

확인 대상:

- 같은 문서 안의 참조
- 같은 폴더의 다른 문서
- `docs/` 하위 README 또는 인덱스 문서
- 프로젝트 루트 문서
- `subscrition/CLAUDE.md` 같은 작업 기준 문서

권장 검색:

```text
기존파일명
기존파일명.md
기존_문서_영문명
```

## 6. 변경 범위를 작게 유지한다

문서 제목이나 파일명 정리 작업에서는 필요한 부분만 수정한다.

- 파일명 변경
- 첫 번째 H1 변경
- 깨진 내부 참조 수정

아래 작업은 별도 요청이 있을 때만 한다.

- 문서 본문 전체 재작성
- 폴더 구조 변경
- 새 인덱스 문서 추가
- 오래된 문서 삭제

## 7. 완료 기준

작업이 끝났다고 보기 전에 아래를 확인한다.

- 파일명이 `현행_` 또는 `이전_`으로 시작한다.
- 첫 번째 H1이 `# [현행]` 또는 `# [이전]`으로 시작한다.
- 코드블록 안의 주석이 제목으로 바뀌지 않았다.
- 기존 파일명 참조가 남아 있지 않다.
- 문서 개수와 폴더 구조가 의도치 않게 바뀌지 않았다.
