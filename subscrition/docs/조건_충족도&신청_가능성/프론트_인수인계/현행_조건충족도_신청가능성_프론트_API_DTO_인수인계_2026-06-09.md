# [현행] 조건 충족도 신청 가능성 프론트 API DTO 인수인계 2026-06-09

<!-- markdownlint-disable MD013 -->

## 1. 문서 목적

이 문서는 프론트 담당자가 “내 조건 충족도 / 신청 가능성 체크리스트” UI를 구현할 때 필요한 controller API와 DTO 필드값을 정리한다.
프론트 구현자는 이 문서만 보고 API 호출, 상태 분기, checklist item 렌더링을 진행할 수 있어야 한다.

## 2. 기능 개요

“내 조건 충족도 / 신청 가능성 체크리스트”는 로그인 사용자의 저장된 프로필과 공고에서 파싱된 자격 조건을 비교해, 공고 상세 화면에서 사용자가 신청 전 확인해야 할 조건을 checklist 형태로 보여주는 기능이다.

이 기능은 최종 신청 가능 여부를 확정하는 기능이 아니다. 사용자가 공고 원문을 보기 전에 “내 프로필 기준으로 충족되는 조건”, “명확히 미충족인 조건”, “원문 또는 프로필 보완 확인이 필요한 조건”을 빠르게 파악하도록 돕는 참고용 기능이다.

프론트에서 만들어야 하는 핵심 UI는 다음과 같다.

| UI 영역 | 설명 |
| - | - |
| 요약 카드 | `summaryStatus`, `summaryMessage`, count 값을 사용해 전체 상태를 보여준다. |
| 조건 checklist | `items` 배열을 사용해 나이, 무주택, 예산, 신청 기간 등 항목별 결과를 보여준다. |
| 상태 badge | `MET`, `NOT_MET`, `NEEDS_VERIFICATION`, `NOT_APPLICABLE`을 색상과 문구로 표시한다. |
| 내 조건 / 공고 조건 비교 | 각 item의 `userValue`, `announcementCondition`을 나란히 보여준다. |
| CTA | `actionLabel`, `actionTarget`에 따라 로그인, 프로필 작성/수정, 공고 원문 확인 동작을 연결한다. |
| disclaimer | API가 내려주는 `disclaimer`를 하단에 표시해 참고용 결과임을 안내한다. |

## 3. 사용자 사용 흐름

기본 사용 흐름:

1. 사용자가 공고 상세 화면에 진입한다.
2. 프론트는 로그인 상태를 확인한다.
3. 비로그인 상태면 checklist API를 호출하지 않고 “내 조건 확인하려면 로그인” CTA를 표시한다.
4. 로그인 상태면 `GET /api/announcements/{announcementId}/eligibility-checklist`를 호출한다.
5. API 응답이 `200`이면 상단 요약 카드와 항목별 checklist를 렌더링한다.
6. 사용자는 각 항목에서 “내 조건”과 “공고 조건”을 비교한다.
7. `NEEDS_VERIFICATION` 항목이 있으면 공고 원문 확인 CTA 또는 프로필 수정 CTA로 유도한다.
8. `NOT_MET` 항목이 있으면 신청 가능성 blocker로 강조하되, 최종 판단은 공고 원문 확인이 필요하다고 안내한다.
9. checklist 하단에는 `disclaimer`를 항상 표시한다.

상태별 예외 흐름:

| 상황 | API 결과 | 프론트 흐름 |
| - | - | - |
| 비로그인 | API 호출 생략 또는 `401 authentication required` | 로그인 CTA 표시 |
| 프로필 미작성 | `400 profile setup required` | 프로필 작성/수정 CTA 표시 |
| 공고가 public-visible 아님 | `404 announcement not found` | checklist 사용 불가 안내 |
| 정상 응답 | `200` | summary card + checklist item list 렌더링 |
| 확인 필요 항목 있음 | `summaryStatus=REVIEW_REQUIRED` 또는 item `NEEDS_VERIFICATION` | 원문 확인/프로필 보완 안내 |
| 미충족 항목 있음 | `summaryStatus=HAS_BLOCKERS` 또는 item `NOT_MET` | blocker 강조, 조건 불일치 이유 표시 |

## 4. Controller API 정보

| 항목 | 값 |
| - | - |
| Controller | `AnnouncementEligibilityChecklistController` |
| Backend file | `subscrition/src/main/java/com/ttait/subscription/announcement/controller/AnnouncementEligibilityChecklistController.java` |
| Base path | `/api/announcements` |
| Method | `GET` |
| Full path | `/api/announcements/{announcementId}/eligibility-checklist` |
| Path variable | `announcementId: number` |
| 인증 | 필요. `Authorization: Bearer {USER_TOKEN}` |
| Request body | 없음 |
| Query string | 없음 |
| Response DTO | `EligibilityChecklistResponse` |
| Response header | `Cache-Control: private, no-store` |

실제 controller 코드 기준:

```java
@RestController
@RequestMapping("/api/announcements")
public class AnnouncementEligibilityChecklistController {

    @GetMapping("/{announcementId}/eligibility-checklist")
    public ResponseEntity<EligibilityChecklistResponse> getEligibilityChecklist(@PathVariable Long announcementId) {
        EligibilityChecklistResponse response = eligibilityChecklistService.getChecklist(CurrentUser.id(), announcementId);
        return ResponseEntity.ok()
                .header(HttpHeaders.CACHE_CONTROL, "private, no-store")
                .body(response);
    }
}
```

주의:

- `SecurityConfig`에서 `GET /api/announcements/**`가 public permit이어도 이 endpoint는 내부에서 `CurrentUser.id()`를 호출한다.
- 따라서 JWT 없이 호출하면 `401 authentication required`가 반환된다.
- 로그인하지 않은 사용자는 프론트에서 API를 호출하지 않고 로그인 CTA를 보여주는 방식이 가장 안전하다.

```http
HTTP/1.1 200 OK

```

fetch 예시:

```ts
async function fetchEligibilityChecklist(announcementId: number, accessToken: string) {
  const response = await fetch(`/api/announcements/${announcementId}/eligibility-checklist`, {
    method: 'GET',
    headers: {
      Authorization: `Bearer ${accessToken}`,
    },
  });

  if (!response.ok) {
    const errorBody = await response.json().catch(() => null);
    throw new Error(errorBody?.message ?? '조건 체크 정보를 불러오지 못했습니다.');
  }

  return response.json() as Promise<EligibilityChecklistResponse>;
}
```

## 10. Enum 값

### 10.1 `EligibilitySummaryStatus`

```ts
export type EligibilitySummaryStatus =
  | 'LIKELY_READY'
  | 'REVIEW_REQUIRED'
  | 'HAS_BLOCKERS'
  | 'INSUFFICIENT_DATA';
```

| 값 | 의미 | UI 권장 |
| - | - | - |
| `LIKELY_READY` | 명확한 미충족/확인 필요 없이 주요 조건 충족 | 긍정 색상, “신청 검토 가능성이 높습니다” |
| `REVIEW_REQUIRED` | 미충족은 없지만 확인 필요 항목 존재 | 주의 색상, “일부 조건 확인이 필요합니다” |
| `HAS_BLOCKERS` | 하나 이상의 미충족 항목 존재 | 위험 색상, “미충족 조건이 있습니다” |
| `INSUFFICIENT_DATA` | 공고 eligibility 또는 판단 데이터 부족 | 중립 색상, “판단 정보가 부족합니다” |

### 10.2 `EligibilityCheckStatus`

```ts
export type EligibilityCheckStatus =
  | 'MET'
  | 'NOT_MET'
  | 'NEEDS_VERIFICATION'
  | 'NOT_APPLICABLE';
```

| 값 | 의미 | UI 권장 |
| - | - | - |

## 12. Error contract

기존 `GlobalExceptionHandler` 형식을 따른다.

```ts
export type ApiError = {
  timestamp: string;
  message: string;
};
```

| 상황 | HTTP status | `message` | 프론트 처리 |
| - | - | - | - |
| JWT 없음 또는 인증 실패 | `401` | `authentication required` | 로그인 CTA 표시 |
| 사용자 profile 없음 | `400` | `profile setup required` | 프로필 작성 CTA 표시 |
| 공고 없음 또는 public-visible 아님 | `404` | `announcement not found` | checklist 영역에 사용 불가 안내 |
| 기타 서버 오류 | `500` | `서버 내부 오류: ...` | 일반 오류 안내 또는 재시도 버튼 |

## 14. 프론트 구현 권장 흐름

1. 공고 상세 페이지에서 로그인 상태를 확인한다.
2. 비로그인 상태면 checklist API를 호출하지 않고 로그인 CTA를 표시한다.
3. 로그인 상태면 `GET /api/announcements/{announcementId}/eligibility-checklist`를 호출한다.
4. loading 중에는 skeleton 또는 로딩 card를 표시한다.
5. `400 profile setup required`면 프로필 작성 CTA를 표시한다.
6. `200`이면 summary card와 item list를 표시한다.
7. `items`는 backend 반환 순서를 그대로 사용하거나 `group`별로 묶는다.
8. `disclaimer`는 checklist 하단에 항상 표시한다.

## 16. 데이터 노출 주의사항

- checklist 응답은 로그인 사용자 프로필 기반 개인화 응답이다.
- backend가 `Cache-Control: private, no-store`를 내려 공용 cache와 브라우저 저장을 방지한다.
- backend는 `eligibilityRaw`, `specialSupplyRaw`, `incomeAssetCriteriaRaw` 원문을 응답하지 않는다.
- `INCOME_ASSETS`는 raw 원문 대신 “공고 원문 확인 필요” 같은 안전 문구만 사용한다.
- 이 API 결과는 최종 신청 가능 확정이 아니라 참고용 checklist다.
