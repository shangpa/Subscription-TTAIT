# 현행 즐겨찾기 일정 MySQL E2E 테스트 기록

작성일: 2026-06-12

## 1. 테스트 목적

실제 Docker Compose MySQL과 local `bootRun` backend를 사용해 `GET /api/me/favorites/schedule` 경로를 검증했다.

검증 범위:

- MySQL container 기동
- Spring Boot backend 기동
- 실제 signup API로 테스트 사용자 생성
- MySQL에 공고, 검수, 즐겨찾기 데이터 적재
- JWT Bearer 인증으로 일정 API 호출
- DB page 정렬 후 200건 cap 동작 확인
- 응답 summary, 그룹 item, calendar event 확인

## 2. 실행 환경

| 항목 | 값 |
|---|---|
| MySQL container | `ttait-subscription-mysql` |
| DB | `ttait_subscription` |
| Backend | local `./gradlew bootRun --args="--spring.task.scheduling.enabled=false"` |
| Base URL | `http://localhost:8080` |
| Test run id | `schedule-e2e-20260612114136` |
| Test user id | `6` |

민감값인 DB password와 JWT accessToken은 문서에 기록하지 않았다.

## 3. 사전 상태와 기동

### 3-1. MySQL 상태

초기 `docker compose ps` 기준 MySQL service가 떠 있지 않았다.

실행:

```bash
docker compose up -d mysql
```

readiness 확인:

```text
mysqld is alive
mysql-ready
```

### 3-2. Backend 상태

`8080` 포트가 비어 있음을 확인한 뒤 backend를 기동했다.

실행:

```bash
./gradlew bootRun --args="--spring.task.scheduling.enabled=false"
```

readiness 확인:

```text
backend-ready
```

테스트 종료 후 backend process는 종료했다. MySQL container와 fixture data는 확인 가능하도록 유지했다.

## 5. API 호출 결과

호출:

```http
GET /api/me/favorites/schedule
Authorization: Bearer {accessToken}
```

핵심 응답 검증:

```text
runId                           totalCount  returnedCount  truncated  firstNoticeName          firstScheduleStatus  groupItemCount  calendarEventCount
schedule-e2e-20260612114136     201         200            true       E2E DUE TODAY PRIORITY  DUE_TODAY            200             600
```

판정:

- HTTP `200` 응답 확인
- `summary.totalCount = 201`
- `summary.returnedCount = 200`
- `summary.truncated = true`
- 첫 번째 그룹 첫 item이 `E2E DUE TODAY PRIORITY`
- 첫 번째 item의 `scheduleStatus = DUE_TODAY`
- 전체 그룹 item 합계는 `200`
- calendar event는 `600`건 생성됨

## 6. 검증 의미

이번 E2E는 아래 경로가 실제로 연결되는지 확인했다.

```text
JWT 인증
→ CurrentUser.id()
→ FavoriteController.schedule()
→ FavoriteScheduleService.getSchedule()
→ UserFavoriteAnnouncementRepository.findSchedulePageByUserIdWithAnnouncement()
→ MySQL page/count query
→ FavoriteScheduleResponse JSON
```

특히 `OPEN` 공고 200건을 먼저 적재하고 `DUE_TODAY` 공고를 나중에 적재했지만, API 응답에서는 `DUE_TODAY` 공고가 첫 번째로 반환됐다. 따라서 저장 순서가 아니라 일정 우선순위 정렬 후 200건 cap이 적용되는 것을 확인했다.
