> 상태: 이전 문서
> 최신 API 계약과 MySQL E2E 결과는 아래 문서를 기준으로 확인한다.
> - `현행_즐겨찾기_일정_DTO_컨트롤러_정리_2026-06-11.md`
> - `현행_즐겨찾기_일정_안정화_변경노트_2026-06-12.md`
> - `현행_즐겨찾기_일정_MySQL_E2E_테스트기록_2026-06-12.md`

> 대상: `subscrition` 백엔드
> 목적: Apidog로 즐겨찾기 일정 API `GET /api/me/favorites/schedule`를 직접 검증한다.

## 1. 사전 준비

백엔드를 먼저 실행한다.

```powershell
cd subscrition
.\gradlew bootRun
```

정상 기준:

```text
Started SubscriptionApplication
Tomcat started on port 8080
```

## 2. Apidog 환경 변수

Apidog Environment에 아래 변수를 만든다.

| 변수명 | 값 | 설명 |
|---|---|---|
| `baseUrl` | `http://localhost:8080` | 로컬 백엔드 주소 |
| `accessToken` | 비워둠 | 로그인 후 저장할 JWT |
| `announcementId` | 비워둠 | 즐겨찾기에 넣을 공고 ID |

로그인이 필요한 API는 Auth 탭에서 `Bearer Token`을 선택하고 값에 `{{accessToken}}`을 넣는다.

## 3. 요청 1: 로그인

| 항목 | 값 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/auth/login` |
| Headers | `Content-Type: application/json` |
| Auth | 없음 |

```json
{
  "loginId": "테스트계정아이디",
  "password": "테스트비밀번호"
}
```

정상 기준:

- HTTP `200`
- 응답에 `accessToken` 존재
- `profileCompleted`, `role` 필드 존재

응답 예시:

```json
{
  "userId": 1,
  "loginId": "user1",
  "accessToken": "eyJhbGciOiJIUzI1NiJ9....",
  "profileCompleted": true,
  "role": "USER"
}
```

로그인 성공 후 `accessToken` 값을 Apidog 환경 변수 `accessToken`에 저장한다.

## 4. 요청 2: 공고 목록에서 announcementId 확보

즐겨찾기 일정 API는 사용자가 저장한 공고가 있어야 의미 있는 응답이 나온다.
먼저 공개 공고 목록에서 하나의 `announcementId`를 확보한다.

| 항목 | 값 |
|---|---|
| Method | `GET` |
| URL | `{{baseUrl}}/api/announcements?page=0&size=20` |
| Auth | 없음 |

정상 기준:

- HTTP `200`
- `content` 배열 존재
- 각 원소에 `announcementId` 존재

응답에서 하나의 `announcementId`를 골라 `announcementId` 환경 변수에 저장한다.

## 5. 요청 3: 즐겨찾기 추가

| 항목 | 값 |
|---|---|
| Method | `POST` |
| URL | `{{baseUrl}}/api/me/favorites` |
| Headers | `Content-Type: application/json` |
| Auth | Bearer `{{accessToken}}` |

```json
{
  "announcementId": {{announcementId}}
}
```

정상 기준:

- HTTP `201`
- 응답 body 없음

이미 즐겨찾기된 공고여도 현재 구현상 오류 없이 지나갈 수 있다.

## 7. 상태값 확인 포인트

`groups[].key`와 `items[].scheduleStatus`는 아래 값 중 하나여야 한다.

- `DUE_TODAY`
- `DUE_TOMORROW`
- `DUE_SOON`
- `OPEN`
- `UPCOMING`
- `DATE_UNKNOWN`
- `CLOSED`

확인 포인트:

- 오늘 마감 공고면 `dDay = 0`, `dDayLabel = D-day`, `scheduleStatus = DUE_TODAY`
- 내일 마감 공고면 `dDay = 1`, `scheduleStatus = DUE_TOMORROW`
- 7일 이내 마감 공고면 `scheduleStatus = DUE_SOON`
- 신청 시작 전 공고면 `scheduleStatus = UPCOMING`
- 신청 시작일 또는 마감일이 없으면 `scheduleStatus = DATE_UNKNOWN`
- 이미 마감된 공고면 `scheduleStatus = CLOSED`, `dDayLabel = 마감`

## 8. 빈 데이터 테스트

즐겨찾기한 공고가 하나도 없는 계정으로 `GET /api/me/favorites/schedule`를 호출하면 아래처럼 내려오는지 확인한다.

- HTTP `200`
- `summary.totalCount = 0`
- `groups`는 존재하지만 각 `items`는 빈 배열
- `calendarEvents`는 빈 배열

## 9. 삭제 동작 확인

일정 API가 즐겨찾기 삭제와 연동되는지 보려면 아래 순서로 확인한다.

### 9-1. 즐겨찾기 삭제

| 항목 | 값 |
|---|---|
| Method | `DELETE` |
| URL | `{{baseUrl}}/api/me/favorites/{{announcementId}}` |
| Auth | Bearer `{{accessToken}}` |

정상 기준:

- HTTP `204`

### 9-2. 일정 API 재호출

다시 `GET {{baseUrl}}/api/me/favorites/schedule`를 호출한다.

정상 기준:

- 방금 삭제한 공고가 `groups[].items`에서 빠짐
- 관련 `calendarEvents`도 함께 사라짐
- `summary.totalCount`가 줄어듦

## 10. 자주 보는 에러

| 상황 | 원인 |
|---|---|
| `401 Unauthorized` | `Authorization: Bearer {{accessToken}}` 누락 또는 만료 |
| `201`은 나오는데 일정 API가 비어 있음 | 즐겨찾기에 넣은 공고가 공개 노출 대상이 아니거나, 다른 계정으로 로그인했을 수 있음 |
| `announcementId`를 못 찾음 | 먼저 `GET /api/announcements`에서 공개 공고를 조회해야 함 |
| `DATE_UNKNOWN`만 나옴 | 해당 공고에 `applicationStartDate` 또는 `applicationEndDate`가 비어 있음 |

## 11. 최소 테스트 순서 요약

1. `POST /api/auth/login`
2. `GET /api/announcements?page=0&size=20`
3. `POST /api/me/favorites`
4. `GET /api/me/favorites/schedule`
5. 필요하면 `DELETE /api/me/favorites/{{announcementId}}`
