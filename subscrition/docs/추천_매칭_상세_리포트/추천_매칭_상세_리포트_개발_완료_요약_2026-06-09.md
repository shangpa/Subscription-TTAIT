# 추천 매칭 상세 리포트 개발 완료 요약 - 2026-06-09

## 개발 범위

- 신규 API: `GET /api/recommendations/{announcementId}/report`
- 목적: 로그인 사용자의 프로필과 단일 공고를 비교해 추천 점수, 요약 상태, 매칭 factor 목록, factor 상태별 count를 반환한다.
- 기존 추천 목록 API와 동일한 산식을 사용하도록 추천 평가 로직을 공통 evaluator로 분리했다.

## 주요 구현

### API

- `RecommendationController`
  - `/api/recommendations/{announcementId}/report` endpoint 추가
  - 개인화 응답이므로 `Cache-Control: private, no-store` 적용

### Service

- `RecommendationReportService`
  - 사용자 프로필 완료 여부 확인

  - 추천 평가 결과를 상세 리포트 응답으로 조립

  - 소득/자산 기준과 세부 자격조건은 원문 확인이 필요한 factor로 분리

## Smoke 중 추가 수정

### 운영 생성자 주입 문제 수정

- `RecommendationMatchEvaluator`의 운영 생성자에 `@Autowired`를 추가했다.
- 테스트용 생성자와 운영 생성자가 함께 있어 Spring bean 생성 시 사용할 생성자를 명확히 지정하기 위한 수정이다.

### LH selected import 재수집 문제 수정

- `AnnouncementUnitRepository.deleteAllByAnnouncementIdInBulk(Long announcementId)` 추가
- `NoticeImportPersistenceService.replaceUnits()`에서 기존 unit row를 bulk delete 후 flush하도록 변경
- `AnnouncementUnitRepository` bulk delete에 `clearAutomatically`, `flushAutomatically` 적용
- bulk delete가 managed entity를 우회하므로 stale `AnnouncementUnit` update를 막기 위한 flush/clear 의도를 repository 주석에 명시했다.
- `AdminLhImportManagementService.importSelected()`의 긴 method-level transaction 제거
  - 외부 LH API, Gemini AI parsing, DB 갱신 전체를 하나의 transaction으로 감싸면서 stale `AnnouncementUnit` flush와 rollback-only 문제가 발생했기 때문이다.
  - import persistence는 기존 `REQUIRES_NEW` transaction 경계를 유지한다.

### 세부 자격조건 확인 factor 보강

- 원 기획의 MVP 범위에 맞춰 `DETAILED_ELIGIBILITY` factor를 추가했다.
- 우선순위, 세대원 조건, 제출서류 등 자동 확정이 어려운 세부 자격조건은 `NEEDS_VERIFICATION`으로 내려준다.
- 원문 텍스트는 응답 DTO에 노출하지 않고 `공고 원문 확인 필요` 값과 원문 CTA만 제공한다.

### 후속 hardening 보강

- public-visible 공고 검수 상태를 `ParseReviewStatus.publicVisibleStatuses()`로 공통화했다.
  - 기존 의미는 `APPROVED`, `CORRECTED` 그대로 유지했다.
  - 공고 조회, 추천 목록, 추천 상세 리포트, 자격 체크리스트, 즐겨찾기, 마감 메일에서 같은 정책을 사용한다.
- selected LH import의 `candidateIds` 입력을 최대 100개로 제한했다.
  - DTO validation: `@Size(min=1, max=100)`
  - controller: `@Valid`
  - service 방어 체크: 초과 시 `400 Bad Request`
