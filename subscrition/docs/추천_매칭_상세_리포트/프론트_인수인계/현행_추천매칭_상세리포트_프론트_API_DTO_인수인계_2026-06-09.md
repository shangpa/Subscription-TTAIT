# [현행] 추천 매칭 상세 리포트 프론트 API DTO 인수인계 2026-06-09

<!-- markdownlint-disable MD013 -->

## 1. 문서 목적

이 문서는 프론트 담당자가 “추천 매칭 상세 리포트” UI를 구현할 때 필요한 backend API, DTO, 상태값, 예외 정책을 정리한다.
프론트 구현자는 이 문서만 보고 추천 카드 내부의 “추천 근거 자세히 보기” 확장 영역 또는 모달을 구현할 수 있어야 한다.

## 2. 기능 개요

추천 매칭 상세 리포트는 기존 맞춤 추천 목록의 `matchScore`, `matchReasons`를 유지하면서, 추천된 공고가 사용자의 프로필과 어떤 항목에서 맞는지 설명하는 참고용 리포트다.

중요한 제한:

- 새 추천 알고리즘이 아니다.
- 당첨 가능성, 신청 가능 여부를 보장하지 않는다.
- 추천 목록에 포함되지 않는 공고는 리포트 대상이 아니다.
- 소득/자산 기준은 자동 확정 판정하지 않고 `NEEDS_VERIFICATION` 또는 `UNKNOWN`으로 안내한다.

## 3. 사용자 사용 흐름

1. 사용자가 `/recommendations` 추천 목록 화면에 진입한다.
2. 프론트는 기존 `GET /api/recommendations?page=&size=`로 추천 카드 목록을 렌더링한다.
3. 사용자가 특정 카드에서 “추천 근거 자세히 보기”를 클릭한다.
4. 프론트는 `GET /api/recommendations/{announcementId}/report`를 호출한다.
5. `200`이면 요약, 기존 추천 사유, factor별 설명, 공고 원문 CTA, disclaimer를 표시한다.
6. `400 profile setup required`이면 프로필 작성/수정 CTA를 표시한다.
7. `404 recommendation not found`이면 추천 대상이 아니거나 현재 사용자에게 리포트를 제공할 수 없는 공고로 안내한다.

실제 controller 코드 기준:

```java
@GetMapping("/{announcementId}/report")
public ResponseEntity<RecommendationReportResponse> getRecommendationReport(@PathVariable Long announcementId) {
    RecommendationReportResponse response = recommendationReportService.getReport(CurrentUser.id(), announcementId);
    return ResponseEntity.ok()
            .header(HttpHeaders.CACHE_CONTROL, "private, no-store")
            .body(response);
}
```

fetch 예시:

```ts
async function fetchRecommendationReport(announcementId: number) {
  const response = await fetch(`/api/recommendations/${announcementId}/report`, {
    method: 'GET',
    headers: {
      Authorization: `Bearer ${accessToken}`,
    },
  });

  if (!response.ok) {
    const errorBody = await response.json().catch(() => null);
    throw new Error(errorBody?.message ?? '추천 근거를 불러오지 못했습니다.');
  }

  return response.json() as Promise<RecommendationReportResponse>;
}
```

## 13. 에러 응답 정책

| 상황 | HTTP | message | 프론트 처리 |
| - | - | - | - |
| 비로그인/JWT 없음 | `401` | `authentication required` | 로그인 CTA |
| 프로필 없음 | `400` | `profile setup required` | 프로필 작성/수정 CTA |
| 공고가 public-visible 아님 | `404` | `announcement not found` | 리포트 사용 불가 안내 |
| public 공고지만 현재 사용자 추천 대상 아님 | `404` | `recommendation not found` | 추천 근거 없음 안내 |

## 14. 프론트 구현 권장

추천 페이지 카드 내부에 아래 순서로 붙이는 방식을 권장한다.

```text
추천 카드
  추천 점수 / 기존 추천 사유
  즐겨찾기 토글
  [추천 근거 자세히 보기]
    로딩 상태
    요약 메시지 + 상태 badge
    기존 추천 사유
    group별 factor list
    [공고 원문 확인]
    disclaimer
```

상태별 UI:

| 상태 | 처리 |
| - | - |
| 접힘 | API 호출하지 않음 |
| 펼침 클릭 | 해당 `announcementId`로 report API 호출 |
| 로딩 | 카드 내부 skeleton 또는 “추천 근거를 불러오는 중입니다” |
| API 실패 | “추천 근거를 불러오지 못했습니다” + 재시도 버튼 |
| `sourceNoticeUrl` 있음 | `OFFICIAL_NOTICE` CTA 또는 하단 원문 링크 표시 |
| `disclaimer` | 항상 표시 |
