# 관리자 기능 구체화

## 현재 구현 범위

관리자 기능은 **AI 파싱 검수**와 **공고 수집 트리거** 두 가지로 구성되어 있음.
사용자 관리나 공고 직접 생성/삭제 같은 일반적인 어드민 CRUD는 미구현 상태.

## API 엔드포인트

모든 `/api/admin/**` 경로는 `ADMIN` 권한 필요 (JWT role claim 기반).

| 메서드 | 경로 | 설명 |
|--------|------|------|
| POST | `/api/admin/import/lh?page=1&size=10` | LH API에서 공고 수집 + PDF AI 파싱 |
| GET | `/api/admin/review/stats` | 검수 통계 대시보드 |

| GET | `/api/admin/review/{announcementId}` | 검수 상세 (AI 파싱 결과 + 원문 비교) |
| POST | `/api/admin/review/{announcementId}` | 검수 액션 처리 |

| POST | `/api/admin/announcements` | 수동 공고 등록 |

## 검수 상태 흐름

```
공고 수집 완료
      │
      ▼
  PENDING ◄─────────────────────────┐
      │                             │
      ├──[APPROVE]──► APPROVED      │
      │                             │
      ├──[CORRECT]──► CORRECTED     │
      │                             │
      ├──[REJECT]───► REJECTED      │
      │                             │
      └──[REIMPORT]─► (재수집+재파싱) ┘
```

### 4가지 검수 액션

| 액션 | 결과 상태 | 설명 |
|------|----------|------|
| `APPROVE` | `APPROVED` | AI 파싱 결과가 정확함 → 그대로 확정 |
| `CORRECT` | `CORRECTED` | AI 오류 필드를 수정 후 확정. `corrections` 객체의 null 필드는 기존 값 유지 |
| `REJECT` | `REJECTED` | AI 품질이 너무 낮거나 공고 자체가 부적절 → 폐기 |
| `REIMPORT` | `PENDING` | LH API + PDF 재수집·재파싱 후 검수 상태 초기화 |

- 흐름
1. APPROVE → reviewStatus = ParseReviewStatus.APPROVED, reviewedBy/ reviewedAt 기록 
2. CORRECT → reviewStatus = ParseReviewStatus.CORRECTED, reviewedBy/reviewedAt/reviewNote 기록
3. REJECT → reviewStatus = ParseReviewStatus.REJECTED, reviewedBy/reviewedAt/reviewNote 기록
4. REIMPORT → 외부 재수집·재파싱 호출 후 reviewStatus = ParseReviewStatus.PENDING으로 초기화

## 권한 구조

### 관리자 계정 생성 방법

회원가입으로 생성된 계정은 기본값 `USER`. DB에서 직접 변경:

```sql
UPDATE users SET role = 'ADMIN' WHERE login_id = '계정ID';
```

서버 재시작 불필요 — 다음 로그인부터 JWT에 `role: ADMIN` 포함.

### JWT → Spring Security 권한 매핑

| JWT role claim | Spring Security 권한 |
|----------------|---------------------|
| `ADMIN` | `ROLE_ADMIN` + `ROLE_USER` |
| `USER` | `ROLE_USER` |

`JwtAuthenticationFilter.resolveAuthorities()` 에서 처리.

## 공고 삭제

`DELETE /api/admin/announcements/{announcementId}`

- `Announcement.softDelete()` 호출 → `deleted=true`, `deletedAt` 기록
- 연관된 `AnnouncementDetail`도 소프트 삭제
- `AnnouncementEligibility`는 soft delete 미지원이나, `findByReviewStatus` 쿼리에서 `announcement.deleted = false` 조건으로 검수 목록에서 자동 제외됨
- 복구 가능 (DB에서 `deleted=false`로 되돌리면 됨)

## 검수 통계 대시보드

`GET /api/admin/review/stats`

### 응답 예시

{

## 버그 수정 이력

### 1. CORRECT 액션 403 반환 (수정 완료)

- **증상**: `POST /api/admin/review/{id}` CORRECT 액션이 항상 HTTP 403 반환
- **원인**: STATELESS 세션 정책에서 예외가 GlobalExceptionHandler를 우회하면 Spring Boot가 `/error`로 포워딩하는데, 인증 컨텍스트가 끊겨 Spring Security가 403 반환
- **수정**:
  - `SecurityConfig.java`: `.requestMatchers("/error").permitAll()` 추가
  - `GlobalExceptionHandler.java`: `@ExceptionHandler(Exception.class)` catch-all 추가 → 500 + 메시지 반환

### 2. 소프트 삭제 공고가 통계 카운트에 포함 (수정 완료)

- **증상**: 소프트 삭제(`deleted=true`)된 공고의 eligibility가 `stats` 상태별 카운트에 계속 포함됨
- **원인**: `AnnouncementEligibilityRepository`의 `countByReviewStatus()`, `countByReviewedAtGreaterThanEqualAndReviewStatusNot()` 파생 메서드가 `announcement.deleted` 조건 미필터
- **수정**: 두 메서드를 `@Query`로 교체, `AND e.announcement.deleted = false` 조건 추가

```java
// AnnouncementEligibilityRepository
@Query("SELECT COUNT(e) FROM AnnouncementEligibility e WHERE e.reviewStatus = :status AND e.announcement.deleted = false")
long countByReviewStatus(@Param("status") ParseReviewStatus status);

@Query("SELECT COUNT(e) FROM AnnouncementEligibility e WHERE e.reviewedAt >= :reviewedAtStart AND e.reviewStatus <> :excludedStatus AND e.announcement.deleted = false")
long countByReviewedAtGreaterThanEqualAndReviewStatusNot(@Param("reviewedAtStart") LocalDateTime reviewedAtStart, @Param("excludedStatus") ParseReviewStatus excludedStatus);
```
