# 현행_내_조건_충족도_신청_가능성_체크리스트_상세기획

- 작성일: 2026-06-02
- 기능명: 내 조건 충족도 / 신청 가능성 체크리스트
- 목적: 공고 상세 페이지에서 사용자의 프로필과 공고 자격조건을 비교해 신청 검토 가능 여부를 항목별로 안내한다.
- 핵심 원칙: 당첨 확률이나 신청 가능 확정이 아니라, 저장된 프로필 기준의 참고용 체크 결과를 제공한다.

## 1. 기능 한 줄 정의

사용자가 공고 상세 페이지를 볼 때, 내 프로필 정보와 공고의 자격조건을 비교해서 **충족 / 미충족 / 확인 필요 / 해당 없음**으로 보여주는 체크리스트 기능이다.

기존 추천 기능이 “이 공고가 맞을 것 같아요”에 가깝다면, 이 기능은 “왜 맞는지, 어디를 조심해야 하는지”를 상세 페이지에서 설명하는 기능이다.

## 2. 사용자에게 보여줄 가치

사용자는 공고를 보고 다음 질문을 바로 해결하고 싶어 한다.

1. 내가 이 공고에 신청할 수 있는가?
2. 내 조건 중 어떤 항목이 맞는가?
3. 어떤 항목은 공고 원문에서 다시 확인해야 하는가?
4. 신청 전에 무엇을 조심해야 하는가?

이 기능은 위 질문에 대해 한눈에 볼 수 있는 판단 보조 UI를 제공한다.

## 3. 표현 원칙

이 기능은 법적/행정적 신청 가능 여부를 확정하지 않는다. 따라서 표현을 조심해야 한다.

| 구분 | 사용하지 않을 표현 | 권장 표현 |
|---|---|---|
| 당첨 가능성 | 당첨 확률 82% | 신청 검토 가능성이 높습니다 |
| 신청 가능 여부 | 신청 가능 확정 | 입력한 프로필 기준으로 조건을 확인했습니다 |
| 자격 보장 | 자격 충족 보장 | 최종 자격은 공고 원문에서 확인해야 합니다 |
| AI 강조 | AI가 당첨 가능성을 예측했습니다 | 저장된 프로필과 파싱된 공고 정보를 기준으로 체크했습니다 |

필수 안내 문구 예시:

```text
이 결과는 저장된 프로필과 파싱된 공고 정보를 기준으로 한 참고용 체크입니다.
최종 신청 가능 여부는 공고 원문과 신청 사이트에서 반드시 확인해야 합니다.
```

## 5. 체크 결과 상태값

각 항목은 아래 4가지 상태 중 하나로 표시한다.

| 상태값 | 화면 표시 | 의미 |
|---|---|---|
| `MET` | 충족 | 현재 프로필 기준 조건을 만족한다. |
| `NOT_MET` | 미충족 | 현재 프로필 기준 조건을 만족하지 못한다. |
| `NEEDS_VERIFICATION` | 확인 필요 | 데이터가 부족하거나 행정적 확인이 필요하다. |
| `NOT_APPLICABLE` | 해당 없음 | 이 공고에는 해당 조건이 명시되지 않았거나 적용 대상이 아니다. |

요약 상태는 아래처럼 둔다.

| 요약 상태 | 화면 표시 | 의미 |
|---|---|---|
| `LIKELY_READY` | 신청 검토 가능성이 높습니다 | 명확한 미충족이 없고 주요 조건 대부분 충족 |
| `REVIEW_REQUIRED` | 일부 조건 확인이 필요합니다 | 미충족은 없지만 확인 필요 항목이 있음 |
| `HAS_BLOCKERS` | 미충족 조건이 있습니다 | 명확한 미충족 항목이 있음 |
| `INSUFFICIENT_DATA` | 판단 정보가 부족합니다 | 공고 또는 프로필 데이터가 부족함 |

## 6. 체크 항목 상세

### 6-1. MVP에서 바로 판정 가능한 항목

| 항목 | 사용자 데이터 | 공고 데이터 | 판정 방식 |
|---|---|---|---|
| 나이 조건 | `profile.age` | `ageMin`, `ageMax` | 나이가 범위 안이면 충족, 벗어나면 미충족 |
| 혼인 상태 | `maritalStatus` | `maritalTargetType` | 미혼/기혼/신혼 등 조건 비교 |
| 신혼 기간 | `marriageYears` | `marriageYearLimit` | 제한 이하이면 충족 |
| 자녀 수 | `childrenCount` | `childrenMinCount` | 최소 자녀 수 이상이면 충족 |
| 무주택 | `homeless` | `homelessRequired` | 필수 조건인데 false면 미충족 |
| 저소득/취약계층 | `lowIncome`, `recipient`, `nearPoverty` | `lowIncomeRequired` | 하나라도 해당하면 충족 |
| 고령자 | `age`, `elderly` | `elderlyRequired`, `elderlyAgeMin` | 기준 나이 이상이면 충족 |
| 보증금 예산 | `maxDeposit` | `depositAmount` | 공고 보증금이 최대 예산 이하이면 충족 |
| 월세 예산 | `maxMonthlyRent` | `monthlyRentAmount` | 공고 월세가 최대 예산 이하이면 충족 |
| 신청 기간 | 현재 날짜 | `applicationStartDate`, `applicationEndDate` | 접수 전/접수 중/마감 상태 안내 |

### 6-2. MVP에서 확인 필요로 처리할 항목

| 항목 | 확인 필요로 두는 이유 |
|---|---|
| 소득 기준 | 공고별 소득 기준이 원문 텍스트 중심이라 숫자 자동 비교가 불안정함 |
| 자산 기준 | 총자산 기준이 공고마다 다르고 정형 필드가 부족함 |
| 차량가액 | 공고별 차량가액 기준이 정형화되어 있지 않음 |
| 거주기간/우선순위 | 현재 별도 구조화 필드가 없음 |
| 제출서류 | 자동 검증 불가 |
| 세대원 전체 조건 | 사용자 본인 입력만으로 확정 불가 |

## 7. 추천 API 구조

공고 상세 API에 끼워 넣기보다 로그인 전용 별도 API로 만드는 것이 좋다.

```http
GET /api/announcements/{announcementId}/eligibility-checklist
```

### 별도 API가 좋은 이유

- 기존 공고 상세 API는 비로그인 사용자도 볼 수 있는 public API다.
- 체크리스트는 로그인 사용자의 프로필이 필요하다.
- public 상세 응답에 사용자별 결과를 섞으면 응답 구조가 복잡해진다.
- 프론트에서 로그인 여부에 따라 별도 호출하기 쉽다.

### 인증/예외 정책

| 상황 | 응답 정책 |
|---|---|
| 로그인 안 됨 | 프론트에서 API 호출하지 않고 로그인 CTA 표시 |
| 프로필 없음 | `400 profile setup required` 또는 별도 상태 응답 |
| 공고가 public 상태 아님 | `404 announcement not found` |
| 자격조건 데이터 없음 | `INSUFFICIENT_DATA` 또는 확인 필요 중심 응답 |
| 정상 | 체크리스트 응답 반환 |

## 8. 백엔드 패키지 구조

추천 구조는 다음과 같다.

```text
subscrition/src/main/java/com/ttait/subscription/announcement/
 ├─ controller/
 │   └─ AnnouncementEligibilityChecklistController.java
 ├─ service/
 │   ├─ EligibilityChecklistService.java
 │   └─ EligibilityCheckEvaluator.java
 └─ dto/
     ├─ EligibilityChecklistResponse.java
     ├─ EligibilityChecklistItemResponse.java
     ├─ EligibilityCheckStatus.java
     └─ EligibilitySummaryStatus.java
```

### 역할 분리

| 파일 | 역할 |
|---|---|
| `AnnouncementEligibilityChecklistController` | HTTP 요청 처리, 현재 사용자 ID 전달 |
| `EligibilityChecklistService` | 공고/프로필/자격조건 조회, public visible 검증, 응답 조립 |
| `EligibilityCheckEvaluator` | 순수 판정 로직 담당 |
| `EligibilityChecklistResponse` | 전체 요약 응답 DTO |
| `EligibilityChecklistItemResponse` | 개별 체크 항목 DTO |
| `EligibilityCheckStatus` | `MET`, `NOT_MET`, `NEEDS_VERIFICATION`, `NOT_APPLICABLE` |
| `EligibilitySummaryStatus` | `LIKELY_READY`, `REVIEW_REQUIRED`, `HAS_BLOCKERS`, `INSUFFICIENT_DATA` |

`RecommendationService` 안의 private 로직을 바로 뜯어내기보다, MVP에서는 `EligibilityCheckEvaluator`를 새로 만들고 테스트를 붙이는 편이 안전하다. 이후 중복이 커지면 공통 evaluator로 통합한다.

## 10. 프론트 화면 구조

대상 화면은 공고 상세 페이지다.

```text
app/src/pages/AnnouncementDetailPage.jsx
```

### 배치 추천

현재 상세 페이지 흐름은 대략 다음과 같다.

```text
공고 제목
신청 기간
기본 정보
위치 정보
시세 비교
비용 정보
공급 정보
공급 단위 정보
안내사항
```

체크리스트는 `신청 기간` 바로 아래, `기본 정보` 위에 배치하는 것이 가장 자연스럽다.

추천 순서:

```text
공고 제목
신청 기간
내 조건 충족도
기본 정보
위치 정보
시세 비교
비용 정보
공급 정보
```

### 우측 sticky card 요약

우측 sticky card에는 간단한 요약만 넣는다.

```text
내 조건 체크
충족 5개 · 확인 필요 2개
일부 조건 확인이 필요합니다
[자세히 보기]
```

### 본문 상세 섹션

본문에는 그룹별 체크리스트를 보여준다.

```text
내 조건 충족도

기본 자격
- 나이 조건: 충족
- 혼인 조건: 확인 필요

가구/상태
- 무주택 조건: 충족
- 자녀 조건: 해당 없음

예산
- 보증금 예산: 충족
- 월세 예산: 충족

원문 확인 필요
- 소득/자산 조건: 확인 필요
- 제출서류: 확인 필요
```

## 12. MVP 범위

### 포함

- 로그인 사용자 전용 체크리스트 API
- 기존 `UserProfile`, `AnnouncementEligibility`, `Announcement` 데이터 재사용
- 나이/혼인/자녀/무주택/저소득/고령자/예산/신청기간 판정
- 소득/자산/서류/거주기간은 확인 필요로 표시
- 공고 상세 페이지에 요약 + 상세 체크리스트 표시
- 과장 표현 방지 안내 문구 포함

### 제외

- 실제 당첨 확률 계산
- AI 기반 확률 예측
- 외부 기관 자격 검증
- 새 DB 테이블
- 소득/자산 기준의 완전 자동 판정
- 제출서류 자동 확인
- 세대원 전체 조건 자동 검증

## 13. 구현 순서

1. 체크 결과 상태값과 DTO 계약 정의
2. `EligibilityCheckEvaluator` 테스트 작성
3. 나이/혼인/자녀/무주택/저소득/고령자/예산/신청기간 판정 구현
4. `EligibilityChecklistService` 구현
5. `AnnouncementEligibilityChecklistController` 구현
6. 공고 상세 프론트에서 로그인 상태일 때 API 호출
7. 체크리스트 UI 추가
8. 비로그인/프로필 없음/로딩/에러 상태 처리
9. 백엔드 테스트 실행
10. 프론트 build 검증
11. 발표용 시나리오 준비

## 14. 테스트 기준

### 백엔드 단위 테스트

| 테스트 | 기대 결과 |
|---|---|
| 나이가 범위 안 | `MET` |
| 나이가 범위 밖 | `NOT_MET` |
| 나이 조건 없음 | `NOT_APPLICABLE` 또는 `NEEDS_VERIFICATION` |
| 무주택 필수 + 사용자 무주택 | `MET` |
| 무주택 필수 + 사용자 유주택 | `NOT_MET` |
| 저소득 필수 + recipient true | `MET` |
| 자녀 최소 수 충족 | `MET` |
| 예산 초과 | `NOT_MET` |
| 소득/자산 원문 존재 | `NEEDS_VERIFICATION` |
| 미충족 항목 존재 | summary `HAS_BLOCKERS` |
| 확인 필요만 존재 | summary `REVIEW_REQUIRED` |

### 프론트 확인

| 상태 | 확인 내용 |
|---|---|
| 비로그인 | 로그인 CTA 표시 |
| 프로필 미작성 | 프로필 설정 CTA 표시 |
| 정상 응답 | 요약/항목 렌더링 |
| `NOT_MET` | 빨간색 blocker 표시 |
| `NEEDS_VERIFICATION` | 노란색 확인 필요 표시 |
| 원문 링크 있음 | 공고 원문 확인 CTA 표시 |

## 15. 발표 시나리오

발표에서는 3가지 케이스를 준비하면 좋다.

| 시나리오 | 설명 | 보여줄 메시지 |
|---|---|---|
| 대부분 충족 | 청년/무주택/예산 조건이 맞는 공고 | 입력한 프로필 기준으로 신청 검토 가능성이 높습니다 |
| 미충족 있음 | 나이 또는 무주택 조건이 맞지 않는 공고 | 미충족 조건이 있어 신청 전 확인이 필요합니다 |
| 확인 필요 많음 | 소득/자산/서류 조건이 많은 공고 | 일부 조건은 공고 원문 확인이 필요합니다 |

## 16. 결론

이 기능은 새로운 거대한 도메인을 만드는 기능이 아니라, 이미 존재하는 추천/프로필/자격조건 데이터를 사용자에게 이해 가능한 형태로 보여주는 기능이다.

3주 안에 구현하려면 범위를 명확히 제한해야 한다.

- 확률 예측은 하지 않는다.
- 신청 가능 확정도 하지 않는다.
- 자동 판정 가능한 조건만 `충족/미충족`으로 표시한다.
- 불확실한 조건은 솔직하게 `확인 필요`로 표시한다.

이렇게 만들면 발표에서는 “청약 따잇은 공고를 보여주는 데서 끝나지 않고, 내 조건 기준으로 신청 전 체크까지 도와준다”는 메시지를 명확하게 보여줄 수 있다.
