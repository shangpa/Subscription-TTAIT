# 현행_새로운_기능_후보_기획

- 작성일: 2026-06-02
- 대상: 청약 따잇(TTAIT) 발표 3주 전 신규 기능 후보
- 전제: `주변시세 비교` 백엔드 구현 완료, 프론트엔드 화면 작업 진행 중
- 목적: 3주 안에 발표 효과가 크고, 현재 코드/데이터를 재사용할 수 있는 신규 기능 후보 선정

## 1. 현재 프로젝트 상태 요약

현재 프로젝트는 단순 공고 목록 서비스보다 이미 넓은 기능 기반을 갖고 있다.

| 영역 | 현재 구현 상태 |
|---|---|
| 공고 수집 | LH 공고 수집, PDF 다운로드, Gemini 기반 AI 파싱, DB 저장 |
| 공고 조회 | 공고 목록/상세, 지역/공급유형/주택유형/기관/상태/가격/카테고리 필터 |
| 사용자 | 회원가입, 로그인, JWT 인증, 프로필 설정/수정 |
| 추천 | 프로필, 카테고리, 자격조건, 예산 기반 맞춤 추천 점수/사유 제공 |
| 즐겨찾기 | 공고 저장, 저장 해제, 저장 목록 조회 |
| 알림 | 추천 이메일, 즐겨찾기 공고 마감 D-7/D-3/D-1 이메일 배치 |
| 위치 | 공고 unit 좌표 저장, 상세 페이지 네이버 지도 표시 |
| 주변시세 | RTMS 원천 수집, snapshot 집계, 공고 unit별 주변시세 비교 API |
| 관리자 | LH 후보 수집, 선택 import, AI 파싱 검수, 승인/수정/반려/재파싱 |

핵심 공백은 기능 기반 부족이 아니라, 사용자가 공고를 보고 **다음 행동을 결정하게 만드는 제품 흐름**이 아직 약하다는 점이다.

## 2. 기능 후보 선정 기준

신규 기능 후보는 아래 기준으로 판단한다.

1. 기존 데이터와 API를 재사용할 수 있는가?
2. 3주 안에 구현 가능한가?
3. 발표에서 한눈에 이해되는가?
4. 서비스 차별점인 “내가 신청 가능한 공고인가?”에 가까운가?
5. 방금 만든 `주변시세 비교` 기능과 충돌하지 않고 흐름을 보완하는가?

## 3. 신규 기능 후보

| 우선순위 | 기능 후보 | 핵심 내용 | 예상 난이도 | 발표 효과 |
|---:|---|---|---|---|
| 1 | 내 조건 충족도 / 신청 가능성 체크리스트 | 공고 상세에서 사용자의 프로필과 공고 자격조건을 비교해 나이, 무주택, 혼인, 자녀, 예산, 소득/자산 조건 충족 여부를 보여준다. | 중 | 매우 높음 |
| 2 | 추천 매칭 상세 리포트 | 추천 목록에서 단순 점수/사유를 넘어서 지역 일치, 예산 적합, 자격 충족, 마감 임박 등을 항목별 breakdown으로 보여준다. | 낮음~중 | 높음 |
| 3 | 즐겨찾기 기반 신청 캘린더 / D-day 관리 | 저장한 공고의 신청 시작일, 마감일, 당첨자 발표일을 개인 일정처럼 보여주고 오늘 확인할 공고를 강조한다. | 낮음~중 | 높음 |
| 4 | 지도형 공고 탐색 | 공고 목록 또는 추천 결과를 지도 마커로 표시하고, 마커 클릭 시 공고 카드/상세로 이동한다. | 중 | 매우 높음 |
| 5 | 주변시세 비교 자동화/default 정책 | 사용자가 기간/sourceType을 몰라도 최근 3~6개월 기준으로 자동 비교하고, 준비 중/데이터 부족 상태를 더 명확히 안내한다. | 중 | 높음 |
| 6 | 마이페이지 알림 센터 + 알림 설정 | 이메일 배치 결과와 마감 알림을 사용자 화면에서 목록/읽음/설정으로 관리한다. | 중 | 중~높음 |
| 7 | 시세 비교 추이/그래프 탭 | 단건 시세 비교를 최근 몇 개월 추이 그래프 또는 snapshot 비교로 확장한다. | 중 | 높음 |
| 8 | 관리자 운영 대시보드 강화 | LH 수집 → AI 파싱 → 검수 → 공개 흐름을 progress timeline과 통계 카드로 보여준다. | 낮음~중 | 중 |

## 4. 1순위 추천: 내 조건 충족도 / 신청 가능성 체크리스트

가장 추천하는 기능은 **내 조건 충족도 / 신청 가능성 체크리스트**다.

이 기능은 원래 기획의 핵심 문장인 “이 공고, 나 신청 가능해?”와 가장 직접적으로 연결된다. 현재 추천 기능은 이미 점수와 추천 사유를 계산하지만, 사용자가 상세 페이지에서 신청 가능 여부를 바로 판단하는 화면은 없다.

### 화면 예시

공고 상세 페이지에 다음과 같은 카드 섹션을 추가한다.

| 항목 | 표시 예시 |
|---|---|
| 나이 조건 | 충족 |
| 무주택 조건 | 충족 |
| 혼인/신혼 조건 | 해당 없음 또는 확인 필요 |
| 자녀 조건 | 충족 / 미충족 |
| 예산 조건 | 보증금/월세 범위 내 |
| 소득/자산 조건 | 확인 필요 |
| 최종 판단 | 신청 가능성이 높은 공고입니다 |

### 장점

- 기존 `UserProfile`, `AnnouncementEligibility`, `RecommendationService` 로직을 재사용할 수 있다.
- 발표에서 서비스 차별점이 바로 드러난다.
- 주변시세 비교와 함께 보여주면 “신청 가능성 + 가격 메리트” 판단 흐름이 완성된다.
- 기능명이 명확해서 팀원 분업과 발표 설명이 쉽다.

### 주의점

- 소득/자산 조건은 현재 raw text 기반인 경우가 많으므로 처음부터 완전 자동 판정으로 만들기 어렵다.
- 1차 구현은 `충족`, `미충족`, `확인 필요` 3단계로 나누는 것이 현실적이다.
- AI가 판단한 것처럼 과장하기보다 “현재 입력한 프로필 기준 체크 결과”로 표현하는 편이 안전하다.

## 5. 3주 발표용 추천 조합

3주 남은 상황에서는 아래 조합이 가장 균형이 좋다.

| 순서 | 기능 | 역할 |
|---:|---|---|
| 1 | 지도형 공고 탐색 | 공고를 발견하는 화면 |
| 2 | 내 조건 충족도 / 신청 가능성 체크리스트 | 신청 가능 여부를 판단하는 화면 |
| 3 | 즐겨찾기 기반 신청 캘린더 / D-day 관리 | 저장 후 행동으로 이어지는 화면 |

이 조합은 사용자 흐름이 자연스럽다.

```text
공고 발견 → 내 조건으로 판단 → 저장 → 신청 일정 관리
```

여기에 이미 구현 중인 `주변시세 비교`를 붙이면 다음 발표 스토리가 된다.

```text
지도에서 공고를 찾고,
내 조건으로 신청 가능성을 확인하고,
주변시세보다 유리한지 비교하고,
즐겨찾기/캘린더로 신청 마감까지 관리한다.
```

## 6. 결론

새 기능을 하나만 고른다면 **내 조건 충족도 / 신청 가능성 체크리스트**를 추천한다.

새 기능 묶음으로 간다면 **지도형 공고 탐색 + 신청 가능성 체크리스트 + 신청 캘린더** 조합이 가장 발표 효과가 크다. 이 조합은 현재 프로젝트의 기존 기능을 버리지 않고, “공고 탐색 서비스”에서 “신청 판단을 도와주는 서비스”로 제품 메시지를 강화한다.
