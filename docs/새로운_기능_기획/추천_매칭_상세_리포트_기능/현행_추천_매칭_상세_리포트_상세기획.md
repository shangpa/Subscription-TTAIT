# 현행_추천_매칭_상세_리포트_상세기획

- 작성일: 2026-06-08
- 기능명: 추천 매칭 상세 리포트
- 목적: 기존 추천 목록의 점수와 추천 사유를 사용자가 이해할 수 있는 항목별 설명 리포트로 확장한다.
- 핵심 원칙: 새 추천 알고리즘을 만드는 기능이 아니라, 현재 추천 결과를 더 투명하게 설명하는 기능이다.

## 1. 기능 한 줄 정의

사용자가 추천 공고를 볼 때, 단순한 점수와 문장형 추천 사유를 넘어서 **지역, 주택 유형, 공급 유형, 예산, 일정, 자격조건** 같은 항목별 근거를 구조화해서 보여주는 상세 리포트 기능이다.

현재 추천 화면이 “이 공고를 추천합니다”에 가깝다면, 이 기능은 “어떤 조건 때문에 추천됐고, 어떤 조건은 확인이 필요한지”를 설명하는 기능이다.

## 2. 사용자에게 보여줄 가치

사용자는 추천 결과를 보고 다음 질문을 바로 해결하고 싶어 한다.

1. 이 공고가 왜 나에게 추천됐는가?
2. 추천 점수는 어떤 요소에서 나온 것인가?
3. 내 프로필과 잘 맞는 조건은 무엇인가?
4. 추천은 됐지만 신청 전에 조심해야 할 항목은 무엇인가?
5. 비슷한 점수의 공고 중 어떤 공고를 먼저 봐야 하는가?

이 기능은 위 질문에 대해 추천 결과의 해석 레이어를 제공한다.

## 3. 표현 원칙

이 기능은 추천 결과를 설명하는 기능이지, 당첨 가능성이나 신청 가능 여부를 보장하는 기능이 아니다. 따라서 표현을 조심해야 한다.

| 구분 | 사용하지 않을 표현 | 권장 표현 |
|---|---|---|
| 당첨 가능성 | 당첨 확률이 높습니다 | 내 프로필과 여러 조건이 잘 맞습니다 |
| 추천 확정성 | AI가 최적 공고로 판단했습니다 | 현재 저장된 프로필과 공고 정보를 기준으로 추천됐습니다 |
| 자격 보장 | 신청 자격을 충족합니다 | 일부 조건은 공고 원문에서 확인해야 합니다 |
| 점수 해석 | 92점이면 당첨 가능 | 92점은 서비스 내부 추천 기준의 일치도입니다 |

필수 안내 문구 예시:

```text
이 리포트는 저장된 프로필과 공고 데이터를 기준으로 추천 결과를 설명하는 참고 자료입니다.
최종 신청 가능 여부와 세부 자격은 공고 원문과 신청 사이트에서 반드시 확인해야 합니다.
```

## 4. 현재 프로젝트에서 재사용 가능한 데이터

### 기존 추천 API

이미 추천 목록 API가 존재한다.

```http
GET /api/recommendations?page=&size=
```

응답은 `Page<RecommendationItemResponse>` 형태다. MVP에서는 이 API의 결과를 기반으로 리포트를 붙이는 방식이 가장 안전하다.

### 기존 추천 응답 DTO

`RecommendationItemResponse`에는 화면과 리포트에서 재사용할 수 있는 필드가 이미 포함되어 있다.

| 필드 | 활용 |
|---|---|
| `announcementId` | 리포트 대상 공고 식별 |
| `noticeName` | 추천 카드와 리포트 제목 |
| `providerName` | 공급 기관 표시 |
| `supplyType` | 공급 유형 매칭 설명 |
| `houseType` | 주택 유형 매칭 설명 |
| `regionLevel1`, `regionLevel2` | 선호 지역 매칭 설명 |
| `fullAddress` | 위치 정보 보조 표시 |
| `complexName` | 단지명 표시 |
| `depositAmount` | 보증금 예산 비교 |
| `monthlyRentAmount` | 월세 예산 비교 |
| `applicationStartDate` | 신청 시작 일정 설명 |
| `applicationEndDate` | 마감 일정과 D-day 설명 |
| `noticeStatus` | 접수 가능 상태 안내 |
| `sourceNoticeUrl` | 공고 원문 확인 CTA |
| `matchScore` | 추천 일치도 요약 |
| `matchReasons` | 기존 추천 사유 문장 표시 및 구조화 후보 |

### 기존 백엔드 파일

후보 #2는 아래 파일과 도메인을 중심으로 설계한다.

| 파일 또는 도메인 | 활용 |
|---|---|
| `RecommendationController` | 추천 목록 API 진입점 |

### 기존 프론트 파일

추천 화면은 이미 존재한다.

```text
app/src/pages/RecommendationsPage.jsx
```

이 파일은 이미 추천 점수, 추천 사유, 즐겨찾기 토글을 렌더링한다. MVP에서는 새 페이지를 크게 만들기보다, 추천 카드의 상세 영역 또는 모달 형태로 리포트를 확장하는 편이 적합하다.

## 5. 리포트 결과 상태값

각 요인은 아래 상태 중 하나로 표시한다.

| 상태값 | 화면 표시 | 의미 |
|---|---|---|
| `STRONG_MATCH` | 잘 맞음 | 사용자 프로필과 공고 조건이 명확히 잘 맞는다. |
| `PARTIAL_MATCH` | 일부 일치 | 큰 방향은 맞지만 세부 조건 확인이 필요하다. |
| `NEEDS_VERIFICATION` | 확인 필요 | 원문 텍스트나 비정형 데이터 때문에 자동 설명이 제한된다. |
| `NOT_MATCHED` | 낮은 일치 | 사용자 선호나 조건과 맞지 않는 부분이 있다. |
| `UNKNOWN` | 정보 부족 | 공고 또는 프로필 데이터가 부족해 설명할 수 없다. |

요약 상태는 아래처럼 둔다.

| 요약 상태 | 화면 표시 | 의미 |
|---|---|---|
| `HIGHLY_RECOMMENDED` | 내 조건과 잘 맞는 추천입니다 | 주요 요인이 대부분 잘 맞음 |
| `RECOMMENDED_WITH_CHECKS` | 추천되지만 확인할 조건이 있습니다 | 추천 점수는 높지만 확인 필요 항목이 있음 |
| `LOW_EXPLANATION_CONFIDENCE` | 추천 사유 설명이 제한적입니다 | `matchReasons` 또는 자격조건 데이터가 부족함 |
| `MIXED_MATCH` | 일부 조건만 맞습니다 | 일치 요인과 불일치 요인이 함께 있음 |

## 6. 리포트 항목 상세

### 6-1. MVP에서 바로 설명 가능한 항목

| 항목 | 사용자 데이터 | 공고 데이터 | 설명 방식 |
|---|---|---|---|
| 지역 | `preferredRegionLevel1`, `preferredRegionLevel2` | `regionLevel1`, `regionLevel2`, `fullAddress` | 선호 지역과 공고 지역의 일치 여부 표시 |
| 주택 유형 | `preferredHouseType` | `houseType` | 선호 주택 유형과 공고 유형 비교 |
| 공급 유형 | `preferredSupplyType` | `supplyType`, `AnnouncementCategory` | 선호 공급 유형과 공고 공급 유형 비교 |
| 보증금 | `maxDeposit` | `depositAmount` | 내 최대 보증금 이내인지 표시 |
| 월세 | `maxMonthlyRent` | `monthlyRentAmount` | 내 최대 월세 이내인지 표시 |
| 신청 일정 | 현재 날짜 | `applicationStartDate`, `applicationEndDate`, `noticeStatus` | 접수 전, 접수 중, 마감 임박, 마감 상태 표시 |
| 추천 사유 | 없음 | `matchReasons` | 기존 문장형 사유를 리포트 상단에 그대로 표시 |

### 6-2. MVP에서 확인 필요로 처리할 항목

| 항목 | 확인 필요로 두는 이유 |
|---|---|
| 소득 기준 | 공고 자격조건이 원문 텍스트 중심이고 정밀 숫자 비교가 어렵다. |
| 자산 기준 | `AnnouncementEligibility`에 원문성 데이터가 포함될 수 있어 완전 자동 설명이 어렵다. |
| 우선순위 조건 | 공고마다 표현이 다르고 별도 구조화 필드가 충분하지 않다. |
| 세대원 조건 | `UserProfile`만으로 확정하기 어렵다. |
| 제출서류 | 추천 근거가 아니라 신청 전 확인 항목으로 다루는 것이 안전하다. |

## 7. API 구조

MVP에서는 기존 추천 목록 API를 유지하고, 상세 리포트 전용 API를 추가하는 방식을 추천한다.

```http
GET /api/recommendations/{announcementId}/report
```

### 별도 API가 좋은 이유

- 기존 `GET /api/recommendations?page=&size=` 응답을 크게 바꾸지 않아도 된다.
- 목록 API의 payload가 과도하게 커지는 것을 막을 수 있다.
- 사용자가 리포트를 펼칠 때만 상세 설명을 조회할 수 있다.
- 기존 `RecommendationsPage.jsx`의 카드 렌더링 구조를 유지할 수 있다.

### 인증/예외 정책

| 상황 | 응답 정책 |
|---|---|
| 로그인 안 됨 | `401` 또는 프론트에서 로그인 CTA 표시 |
| 프로필 없음 | 리포트 대신 프로필 작성 CTA 표시 |
| 추천 대상 공고 없음 | `404 recommendation not found` |
| 공고가 public 상태 아님 | `404 announcement not found` |
| factor 설명 불가 | `LOW_EXPLANATION_CONFIDENCE`와 확인 필요 항목 중심 응답 |
| 정상 | 추천 상세 리포트 응답 반환 |

## 8. 백엔드 패키지 구조

기존 추천 API가 있는 `announcement` 패키지를 유지하는 구조를 추천한다.

```text
subscrition/src/main/java/com/ttait/subscription/announcement/
 ├─ controller/
 │   └─ RecommendationController.java
 ├─ service/
 │   ├─ RecommendationService.java
 │   └─ RecommendationReportService.java
 └─ dto/
     ├─ RecommendationItemResponse.java
     ├─ RecommendationReportResponse.java
     ├─ RecommendationFactorResponse.java
     ├─ RecommendationFactorStatus.java
     └─ RecommendationReportSummaryStatus.java
```

### 역할 분리

| 파일 | 역할 |
|---|---|
| `RecommendationController` | 기존 추천 목록 API와 상세 리포트 API 제공 |
| `RecommendationService` | 기존 추천 목록, 점수, 사유 생성 흐름 유지 |
| `RecommendationReportService` | 사용자 프로필과 추천 공고를 조회해 factor별 설명 조립 |
| `RecommendationReportResponse` | 전체 요약, 점수, factor 목록, 안내 문구 반환 |
| `RecommendationFactorResponse` | 개별 factor의 상태와 설명 반환 |
| `RecommendationFactorStatus` | `STRONG_MATCH`, `PARTIAL_MATCH`, `NEEDS_VERIFICATION`, `NOT_MATCHED`, `UNKNOWN` |
| `RecommendationReportSummaryStatus` | 리포트 요약 상태 enum |

`RecommendationService`의 기존 추천 계산을 크게 바꾸지 않는다. MVP에서는 상세 리포트 조립 책임을 `RecommendationReportService`로 분리하고, 기존 `matchScore`, `matchReasons`를 그대로 신뢰한다.

## 10. 프론트 화면 구조

대상 화면은 추천 페이지다.

```text
app/src/pages/RecommendationsPage.jsx
```

### 배치 추천

현재 추천 페이지는 이미 점수, 추천 사유, 즐겨찾기 토글을 보여준다. 상세 리포트는 카드 안에 기본 요약을 보여주고, 버튼을 누르면 확장 영역 또는 모달로 자세히 보여주는 방식이 적합하다.

추천 순서:

```text
추천 카드
추천 점수
기존 추천 사유
즐겨찾기 토글
추천 근거 자세히 보기
```

확장 영역 예시:

```text
추천 매칭 리포트

요약
지역, 유형, 예산 조건이 잘 맞습니다.

선호 조건
- 지역: 잘 맞음
- 주택 유형: 잘 맞음
- 공급 유형: 일부 일치

비용
- 보증금: 잘 맞음
- 월세: 잘 맞음

신청 전 확인
- 소득/자산 기준: 확인 필요
- 세부 자격조건: 확인 필요

[공고 원문 확인]
```

### 점수 표시 방식

`matchScore`는 계속 보여주되, 점수를 과하게 해석하지 않도록 문구를 붙인다.

```text
추천 일치도 88점
서비스 내부 기준으로 내 프로필과 공고 정보가 얼마나 맞는지 나타냅니다.
```

| 리포트 접힘 | 기존 추천 카드만 표시 |
| 리포트 로딩 | 카드 내부 skeleton 또는 로딩 텍스트 표시 |
| 리포트 API 에러 | “추천 근거를 불러오지 못했습니다” 표시 |
| 설명 신뢰도 낮음 | `matchReasons`와 원문 확인 CTA 중심으로 표시 |
| 정상 | factor별 리포트 표시 |

## 12. MVP 범위

- 기존 `RecommendationItemResponse`의 `matchScore`, `matchReasons` 재사용

- 공고 원문 확인 CTA와 참고용 안내 문구 포함

### 제외

- 새 추천 알고리즘 개발

## 14. 테스트 기준

### 백엔드 단위 테스트

| 테스트 | 기대 결과 |
|---|---|
| 선호 지역과 공고 지역 일치 | 지역 factor `STRONG_MATCH` |
| 선호 지역과 공고 지역 불일치 | 지역 factor `NOT_MATCHED` 또는 `PARTIAL_MATCH` |
| 선호 주택 유형 일치 | 주택 유형 factor `STRONG_MATCH` |
| 보증금이 예산 이하 | 보증금 factor `STRONG_MATCH` |
| 보증금이 예산 초과 | 보증금 factor `NOT_MATCHED` |
| 월세 정보 없음 | 월세 factor `UNKNOWN` |
| 신청 마감일이 가까움 | 일정 factor에 마감 임박 메시지 포함 |
| 소득/자산 원문 존재 | 소득/자산 factor `NEEDS_VERIFICATION` |
| `matchReasons`가 비어 있음 | summary `LOW_EXPLANATION_CONFIDENCE` 가능 |
| 확인 필요 factor 존재 | summary `RECOMMENDED_WITH_CHECKS` 가능 |

### 프론트 확인

| 상태 | 확인 내용 |
|---|---|
| 추천 카드 기본 상태 | 기존 점수와 추천 사유가 유지됨 |
| 자세히 보기 클릭 | 리포트 영역 또는 모달 표시 |
| 로딩 중 | 리포트 로딩 상태 표시 |
| API 실패 | 리포트 오류 메시지 표시 |
| 확인 필요 factor | 노란색 또는 주의 스타일로 표시 |
| 공고 원문 URL 있음 | 원문 확인 CTA 표시 |
| 즐겨찾기 토글 | 기존 동작과 충돌하지 않음 |

## 15. 발표 시나리오

발표에서는 3가지 케이스를 준비하면 좋다.

| 시나리오 | 설명 | 보여줄 메시지 |
|---|---|---|
| 추천 근거가 명확한 공고 | 지역, 유형, 예산이 모두 잘 맞는 공고 | 내 조건과 잘 맞는 추천입니다 |
| 확인 필요가 있는 공고 | 추천 점수는 높지만 소득/자산 확인이 필요한 공고 | 추천되지만 확인할 조건이 있습니다 |
| 설명 데이터가 부족한 공고 | `matchReasons`가 적거나 자격조건 구조화가 부족한 공고 | 추천 사유 설명이 제한적입니다 |

## 16. 리스크와 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| `matchReasons`가 문자열 배열이다 | factor별 정밀 설명으로 바로 변환하기 어렵다 | MVP에서는 원문 사유를 그대로 보여주고, 새 factor DTO는 별도 조립한다 |
| factor 수준 DTO가 아직 구조화되어 있지 않다 | 프론트와 백엔드 계약이 새로 필요하다 | `RecommendationFactorResponse`를 작게 정의한다 |
| 자격조건 원문 텍스트가 많다 | 소득/자산 설명의 정확도가 낮아질 수 있다 | 자동 판정 대신 확인 필요로 표시한다 |
| backend와 frontend DTO가 어긋날 수 있다 | 화면 오류가 발생할 수 있다 | API 예시를 먼저 고정하고 프론트 렌더링을 맞춘다 |
| 추천 점수 해석이 과장될 수 있다 | 사용자가 당첨 가능성으로 오해할 수 있다 | 안내 문구와 표현 원칙을 화면에 포함한다 |

## 17. 결론

이 기능은 추천 알고리즘을 새로 만드는 기능이 아니라, 이미 존재하는 추천 결과를 사용자에게 이해 가능한 리포트로 보여주는 기능이다.

3주 안에 구현하려면 범위를 명확히 제한해야 한다.

- 기존 추천 점수와 추천 사유를 유지한다.
- factor 설명은 정형 데이터로 가능한 항목만 다룬다.
- 소득, 자산, 세부 자격조건은 확인 필요로 둔다.
- 당첨 확률이나 신청 가능 확정 표현은 쓰지 않는다.

이렇게 만들면 발표에서는 “청약 따잇은 추천만 던져주는 것이 아니라, 추천된 이유까지 설명해 사용자가 더 안심하고 비교할 수 있게 한다”는 메시지를 명확하게 보여줄 수 있다.
