# 공고 필터링 수정 정리

작성일: 2026-05-06

## 1. 수정 배경

이번 수정은 공고 목록 필터에서 아래 두 문제가 확인되어 진행했다.

- `시/군/구` 필터가 정상 동작하지 않음
- `주택유형` 필터 옵션이 비어 있거나 필터가 제대로 걸리지 않음

## 2. 원인 정리

### 2.1 시/군/구

원인:

- `GET /api/filters/regions/level2` 가 상위 지역 파라미터를 실제로 받지 않고 있었음
- 하위 지역 목록 생성 시 `regionLevel1` 기준 필터링이 없었음
- 일부 데이터는 `region_level2` 컬럼이 비어 있고 `full_address` 에만 `수원시`, `강릉시` 같은 값이 들어 있었음

영향:

- 프론트에서 `경기도`, `강원도`를 선택해도 하위 지역 목록이 정확히 내려오지 않음
- 공고 조회 시 `regionLevel2` 를 선택해도 실제 데이터와 매칭되지 않는 케이스가 발생함

### 2.2 주택유형

원인:

- 기존 `houseTypeOptions()` 는 `announcement.houseTypeNormalized` 값만 기준으로 옵션을 생성했음
- 실제 DB에는 `houseTypeNormalized` 가 비어 있고 `houseTypeRaw` 만 존재하는 데이터가 있을 수 있음

영향:

- `GET /api/filters/house-types` 응답이 비거나 일부 유형이 누락될 수 있음
- `houseType` 필터도 `houseTypeNormalized` 값이 없는 공고는 매칭되지 않음

## 3. 수정 내용

### 3.1 지역 필터 수정

수정 파일:

- `subscrition/src/main/java/com/ttait/subscription/announcement/controller/FilterController.java`
- `subscrition/src/main/java/com/ttait/subscription/announcement/repository/AnnouncementRepository.java`
- `subscrition/src/main/java/com/ttait/subscription/announcement/service/AnnouncementQueryService.java`

수정 사항:

- `/api/filters/regions/level2` 에서 `level1` 또는 `regionLevel1` 파라미터를 받을 수 있도록 변경
- `regionLevel1` 기준으로 활성 공고를 조회해 하위 지역 옵션 목록을 생성하도록 변경
- `region_level2` 값이 비어 있으면 `full_address` 에서 `시/군/구` 를 추출해 fallback 하도록 변경
- 공고 목록 응답에서도 `regionLevel2` 는 보정된 값이 내려가도록 변경
- 공고 조회 필터에서도 `regionLevel2` 비교 시 보정된 값을 기준으로 매칭하도록 변경

예:

- `경기도 수원시 권선구 ...` -> `수원시`
- `강원도 강릉시 ...` -> `강릉시`
- `성남시 분당구` -> `성남시`

### 3.2 주택유형 필터 수정

수정 파일:

- `subscrition/src/main/java/com/ttait/subscription/announcement/service/AnnouncementQueryService.java`
- `subscrition/src/main/java/com/ttait/subscription/announcement/service/RecommendationService.java`

수정 사항:

- `houseTypeOptions()` 가 더 이상 `houseTypeNormalized` 만 보지 않도록 변경
- 공고별로 아래 우선순위로 주택유형을 해석하도록 변경

우선순위:

1. `houseTypeRaw` 가 있으면 `AnnouncementNormalizer.normalizeHouseType()` 로 정규화
2. 없으면 `houseTypeNormalized` 사용

적용 범위:

- 주택유형 옵션 목록 생성
- 공고 목록 필터링
- 공고 목록 응답 DTO
- 공고 상세 응답 DTO
- 추천 공고 점수 계산 및 추천 응답 DTO

효과:

- DB의 `houseTypeNormalized` 값이 비어 있어도 `houseTypeRaw` 만 있으면 옵션이 보이고 필터도 동작함

### 3.3 추천 로직 정합성 보정

수정 파일:

- `subscrition/src/main/java/com/ttait/subscription/announcement/service/RecommendationService.java`

수정 사항:

- 추천 점수 계산 시 `preferredRegionLevel2` 비교도 보정된 `regionLevel2` 값 기준으로 변경
- 추천 점수 계산 시 `preferredHouseType` 비교도 fallback 된 주택유형 기준으로 변경
- 추천 응답 DTO 에도 보정된 `regionLevel2`, `houseType` 값을 내려주도록 변경

## 4. 테스트 코드 추가

추가 파일:

- `subscrition/src/test/java/com/ttait/subscription/announcement/service/AnnouncementQueryServiceTest.java`

테스트 대상:

- `regionLevel2Options()` 가 상위 지역 기준으로 하위 지역 목록을 생성하는지
- `fullAddress` 만 있어도 `regionLevel2` 필터가 동작하는지
- `houseTypeRaw` 로부터 주택유형 옵션이 생성되는지
- `houseTypeRaw` 기반 fallback 으로 `houseType` 필터가 동작하는지

## 5. Apidog 확인 예시

### 5.1 시/군/구 옵션 조회

```http
GET /api/filters/regions/level2?level1=경기도
```

또는

```http
GET /api/filters/regions/level2?regionLevel1=경기도
```

기대:

- 응답의 `items` 에 `수원시`, `성남시` 같은 하위 지역이 포함되어야 함

### 5.2 시/군/구 필터 조회

```http
GET /api/announcements?regionLevel1=경기도&regionLevel2=수원시&page=0&size=20
```

기대:

- `fullAddress` 가 `경기도 수원시 ...` 인 공고만 조회되어야 함

### 5.3 주택유형 옵션 조회

```http
GET /api/filters/house-types
```

기대:

- `houseTypeNormalized` 가 비어 있어도 `houseTypeRaw` 기반으로 `아파트`, `다세대/연립`, `오피스텔` 등이 보여야 함

### 5.4 주택유형 필터 조회

```http
GET /api/announcements?houseType=아파트&page=0&size=20
```

기대:

- `houseTypeRaw` 가 `아파트형 공공주택` 같은 값이어도 `아파트` 로 정규화되어 조회되어야 함

## 6. 같이 반영한 운영 편의 수정

수정 파일:

- `.gitignore`

추가한 항목:

- `env`
- `**/application-local.properties`
- `**/application-*.local.properties`

목적:

- 로컬 시크릿 파일이 Git 에 실수로 올라가지 않도록 방지

주의:

- 이미 Git 이 추적 중인 파일은 `.gitignore` 만으로 제거되지 않음
- 이미 노출된 키는 반드시 재발급이 필요함

## 7. 참고 사항

- 작업 중 `subscrition/src/main/java/com/ttait/subscription/admin/dto/ManualAnnouncementRequest.java`
  와 `subscrition/src/main/resources/application-local.properties` 는 기존 변경이 있었으며, 이번 필터링 수정과 직접 관련된 부분은 건드리지 않았다.
- 이 환경에서는 Gradle 테스트를 직접 실행하지 못해, 테스트 코드는 추가했지만 실행 결과까지 문서화하지는 않았다.
