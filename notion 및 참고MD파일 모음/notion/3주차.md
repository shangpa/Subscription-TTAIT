### ai 가 만들어준 기초 토대 만들기 *(from. 김재민)*

수정하신 기능 리스트(PDF+AI 파싱 전략, 지도 마커, 실거래가 XML 등)를 반영하여, 정식 버전에서 어디부터 어디까지 만들어야 하고, 데모의 단위 기능들을 어떻게 끼워 넣게 판을 짤지 팀원이 AI에게 전달하기 좋은 프롬프트로 다시 작성했습니다.

코드나 패키지 이야기는 빼고 철저히 **기능 범위와 확장성(인터페이스/DB 설계)**에 초점을 맞추었습니다. 아래 내용을 복사해서 팀원에게 전달해 주시면 됩니다.

📋 [프롬프트 내용]
[상황 및 역할 부여] 너는 지금부터 "청약 따잇(청년·신혼부부·무주택자·고령자·저소득자 대상 공공임대 맞춤형 공고 알림 서비스)"의 **정식 버전 기능적 토대(Foundation)**를 구축하는 수석 백엔드 개발자이다.

현재 별도의 '데모 버전'에서 복잡한 외부 API 연동, PDF 다운로드 및 AI 문자 추출, 외부 실거래가 API(XML) 파싱 등 고난도 비즈니스 로직을 샌드박스로 테스트하고 있다. 너의 역할은 데모 버전에서 완성된 이 복잡한 기능들이 나중에 정식 버전에 '플러그인'처럼 쉽게 끼워 맞춰질 수 있도록, 정식 버전의 전체적인 서비스 흐름(기본 CRUD, DB 스키마, API 뼈대, 인터페이스 경계)을 미리 세팅해 두는 것이다.

아래 명시된 **[구현 범위]**를 읽고, 각 기능별로 어디부터 어디까지 만들어야 하는지, 그리고 나중에 확장을 위해 어떤 구조(인터페이스, DB 필드, DTO 응답 포맷)를 고려해야 하는지 설계해 줘.

1. 공고 데이터 수집 및 저장 (관리자 기능 - PDF 및 AI 파싱 포함)
    
    [어디부터 어디까지 만들어야 하는가]
    
    - 어디부터: 시스템에 공고 데이터를 저장, 수정, 조회할 수 있는 내부 DB 스키마(공고 모델) 설계와 기본 CRUD API 구축.
    - 어디까지: "외부 API에서 받아온 JSON 데이터"와 "AI가 PDF를 파싱한 결과 데이터"가 합쳐서(Merge) 넘어왔을 때, 이를 DB에 예쁘게 저장할 수 있는 '수신용 API 엔드포인트(혹은 서비스 로직)' 까지만 만든다.
    - 제외할 부분(데모에서 가져올 것): 외부 API를 직접 호출하거나, PDF를 다운로드하고 AI API를 태워 텍스트를 파싱하는 로직. 외부 평면도 웹 크롤링 로직.
    
    💡 [이후 확장을 위한 고려사항]
    
    데모 버전의 파싱 로직이 붙을 것을 대비해, 파싱 방식을 유연하게 선택할 수 있는 구조(NoticeParsingStrategy 같은 인터페이스)를 미리 뚫어둔다.
    
    DB에는 원본 JSON 데이터, 원본 PDF 링크, 그리고 **AI가 추출한 정제된 텍스트(ai_parsed_data)**를 따로 보관할 수 있는 칼럼을 넉넉하게(JSON/TEXT 타입 등으로) 만들어 둔다.
    
2. 회원가입 및 프로필 관리 (사용자 기능)
    
    [어디부터 어디까지 만들어야 하는가]

    - 어디까지: 기획에 정의된 사용자 조건(신분 유형: 청년/신혼부부 등, 가구원수, 취약계층 여부, 소득액, 자산, 차량가액 등)을 모두 담을 수 있는 DB 스키마와 유저 엔티티 설계.
    
    💡 [이후 확장을 위한 고려사항]
    
    사용자가 입력한 나이, 소득, 자산 데이터는 나중에 "AI 당첨 가능성 리포트"에서 가점 계산의 핵심 변수로 쓰인다.
    
    단순 텍스트가 아니라 계산이 가능한 숫자형(Integer, Long)이나 명확한 코드로 정형화해서 저장해야 한다.
    
3. 맞춤형 공고 추천 리스트 & 지도 위치 표시
